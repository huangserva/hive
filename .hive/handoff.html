2.5 分钟上手指南

Step 1创建飞书自建应用（一次性）
  1. 打开 open.feishu.cn → 我的应用 → 创建自建应用
  2. 填名字（如 "hive-orchestrator"）+ 图标，提交
  3. 左侧 添加应用能力 → 机器人，开启
  4. 左侧 权限管理，添加：
    • im:message
    • im:message:send_as_bot
    • im:resource
    • im:chat
  5. 左侧 事件订阅 → 订阅模式选 WebSocket（不要选回调，不需要公网 IP）
  6. 订阅事件：搜索 im.message.receive_v1 → 添加
  7. 左侧 凭证与基础信息，记下 App ID（cli_xxxxx）和 App Secret
  8. 右上角 版本管理与发布 → 发布应用（先创建一个 1.0 版本）
Step 2写凭证文件
cat > ~/.config/hive/feishu.json <<'EOF'
{
  "app_id": "cli_你的app_id",
  "app_secret": "你的app_secret"
}
EOF

文件不在则 hive 启动时 log "feishu credentials not configured, transport disabled"——优雅降级，不影响其他功能。

Step 3重启 4010
# 杀当前 dev runtime（注意：这会杀掉所有 worker PTY）
# 在另一个终端窗口（不是 orchestrator PTY 里）跑：
lsof -ti:4010 | xargs kill -9 2>/dev/null
cd ~/development/hive-serva
pnpm dev

重启后看 ~/.config/hive/logs/runtime-4010.log，应该出现：

[info] feishu credentials loaded app_id=cli_xxxxx
[info] runtime started port=4010 ...
[info] feishu WSClient connected
Step 4在 hive web UI 绑定 chat
  1. 打开 http://127.0.0.1:5173
  2. 顶栏右侧应该看到一个绿色 🪶 指示灯（鼠标悬停看 tooltip）
  3. 侧边栏 active workspace 行右侧有个 ⚙ 设置图标 → 点开
  4. 弹出 "Feishu Integration" dialog → 输入 chat_id（在飞书群里 @bot 一次后看 hive log 能拿到 oc_xxx）+ 可选 chat 名 → Bind

第一次拿 chat_id 最简单的方法：把 bot 拉进群（或私聊），@它说一句话，看 hive log 里会输出 feishu inbound dropped reason=no_binding chat_id=oc_xxxxx——那个 oc_xxxxx 就是 chat_id。

Step 5飞书发消息测试 e2e
  1. 在飞书群 @bot 说："今天天气怎么样？"（或任何 orchestrator 能 handle 的话）
  2. hive web UI 上 orchestrator PTY 应该收到注入：[来自飞书 chat=oc_xxx，sender=ou_xxx user_id=ou_xxx]\n请用 team feishu reply 回复...\n---\n今天天气怎么样？
  3. orchestrator 处理 + 派 worker（如果需要）+ 用 team feishu reply "回复内容" 回话
  4. 飞书群里应该立刻收到 bot 的回复消息
  5. 测审批：飞书 @bot 说"删除 ~/Downloads/old"——orchestrator 应该先调 team approve，飞书发审批卡片，你点 ❌ → 卡片变灰 + 飞书收到"已撤销"。再试一次点 ✅ → 卡片变绿 + worker 真去删

2.5.审批卡片流程（Hermes 风格）

触发时机：orchestrator 收到飞书消息 + 准备做高风险动作（rm / git push / drop / 删大量文件 / 写外部服务 / 任何不可逆操作），会自觉调 team approve "动作描述" --risk high。

飞书侧收到的卡片：

┌──────────────────────────────────┐
│ 🟥 🤖 Hive 审批请求 · 高风险动作   │
├──────────────────────────────────┤
│ 动作                              │
│ 派关羽删除 ~/Downloads/old 目录    │
│                                  │
│ 派给           Workspace          │
│ 关羽 (codex)    hive-serva        │
├──────────────────────────────────┤
│   [✅ 允许]    [❌ 拒绝]            │
└──────────────────────────────────┘

点 ✅：卡片变绿色只读「✅ 已允许 by @huangserva at 16:42」+ orchestrator stdin 收到 [Hive 系统消息：approval_id=xxx ALLOWED by feishu user_id=ou_xxx at 16:42:08] → 继续派单。

点 ❌：卡片变灰色「❌ 已拒绝」+ orchestrator 收到 DENIED → 自动 team feishu reply "已撤销，请提供替代方案"。

medium 风险（如 git status / 改配置文件）：橙色 header「中风险动作」，按钮一样。

低风险（查 log / 跑测试 / git status）：不弹卡片，直接派单（orchestrator 自行判断）。

No ACL：群里任何人点都算数（单人用场景）。Runtime 重启后未处理的 pending 卡片点击会显示「审批已处理 / 已过期」。判断动作风险等级 100% 靠 orchestrator system prompt 引导，不做关键词拦截——LLM 比 denylist 更鲁棒。

限制影响缓解 / 未来方向

群聊 mention 判断只看 mentions 数组非空 群里其他 @bot（多 bot 场景）也会触发 未来加 bot_open_id 严格校验

sender 展示名是 user_id 不是真名 orch 收到的注入文本里 sender=ou_xxx 而不是"张三" 未来调 contact API 拿展示名 + cache
TaskLog/对话历史 UI 没飞书标记 web 上看不出哪条消息来自飞书 需要先有 persistent message log 面板，再加 sender badge
FeishuTransport 状态机无单测 未测重连状态切换正确性 关羽建议加 testing seam（protected setter 或 event emitter）
WorkspaceSettings dialog 无 React test UI 改动可能默默 break 用 Testing Library 加 component 测试
飞书消息频率限制 200/min/app 单人用不会触发，但 outbox 没节流 未来加 token bucket 节流
orchestrator 离线时飞书消息丢 没排队机制，bot 立刻回 "Orchestrator 未运行" TTL 排队（Phase 1 设计中 Option B 留了空间）
UTF-8 BOM 的 feishu.json 无法 parse Windows 编辑器存的文件可能含 BOM 而导致启动失败 typewriter 已有测试 lock 当前行为；想支持就在 feishu-credentials.ts 加一行 strip BOM
