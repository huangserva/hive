# 决策：M14 语音/移动路线先走 Feishu voice command MVP

**日期**: 2026-05-25
**状态**: draft（待 user 确认）
**关联**: plan.md → M14 mobile + voice 扩张方向

## 背景

user 已确认 M14 是未来方向，北极星是“语音控制多 agent 开发”。M14 开工前需要在三条路线中选第一步：

1. 自建 mobile app：iOS/Android/Expo + paseo expo-two-way-audio 类双向音频模块。
2. 借第三方 voice agent framework：LiveKit / Pipecat / Vapi / OpenAI Realtime 等。
3. 飞书 + voice plugin：复用现有 Feishu bridge，在飞书侧加语音消息/语音转文字入口。

这不是最终产品形态裁决，而是 M14 第一阶段怎么进入。

## 决策

推荐 **先采用 Feishu voice command MVP**：

- 最小形态：用户在飞书移动端发语音或语音+文字；runtime 取 transcript（优先 Feishu 自带转写，fallback 到 ASR provider）；再复用现有 Feishu inbound prompt 注入 orchestrator。
- orch 继续使用既有 PM/team protocol：`team send` 派 worker、`team approve` 做高风险审批、`team feishu reply` 回用户。
- 暂不做实时双向音频，不新建 mobile app，不默认引入 Vapi 等付费托管平台。
- 保留升级出口：如果 Feishu voice command 被高频使用且用户明确需要连续对话/打断/低延迟播报，再做 LiveKit/OpenAI Realtime/self-built mobile 的第二阶段 spike。

## 理由

1. **最先验证核心价值**：语音控制多 agent 的第一步是“口述任务能否被 PM 正确转成 dispatch/approval/report 闭环”，不需要先实现实时通话体验。
2. **复用现有 M4 Feishu bridge**：workspace binding、WS inbound、outbound reply、approval card、runtime credentials/logging 都已存在，改动最小。
3. **保留移动场景**：Feishu mobile 已经是手机入口，能直接验证“离开桌面时控制 Hive”。
4. **降低媒体工程风险**：自建 app 会提前引入 app 分发、移动权限、真机音频、AEC、后台、remote auth；这些是未来上限，不适合第一 slice。
5. **避免过早 vendor lock-in**：Vapi 类托管平台需要预算和数据策略拍板；LiveKit/Pipecat 更适合 realtime 阶段，不应为 voice command MVP 建第二套 runtime。

## 已知代价

- Feishu route 不是 realtime conversation；不能边听边打断，也没有低延迟语音回放。
- 依赖 Feishu 客户端与 bot audio event 能力；需要 M14a 真 Feishu E2E 验证 payload、下载权限、转写可用性。
- 如果用户最终想要品牌化独立 mobile app，这只是第一阶段，不是终局。

## 备选方案

- **直接自建 mobile app**：长期上限最高，但 4-8 周级别，第一阶段 ROI 低，容易先卡在 mobile/audio infra。
- **LiveKit first**：适合 realtime voice agent，但会引入 room/agent server/job lifecycle，需要谨慎映射到 HippoTeam orchestrator+PTY。
- **Vapi first**：原型最快但托管付费、数据策略和成本不可默认，需要 user 另行确认。

## 结果（后写）

待 user 确认后，M14a 可拆为 Feishu voice event spike、audio download/transcript provider、inbound prompt injection、真实飞书 E2E、可选 voice reply。
