# 决策：实时通话「投机式前台 + 客户端播放闸 + GLM 完整意图交 PM」架构

**日期**: 2026-06-06
**状态**: 已采纳（user 2026-06-06 深夜拍板"这个做成决议吧"）
**关联**: plan.md → M40（实时通话理解层）；承接 M39（流式 ASR）、ADR `draft-2026-06-05-realtime-voice-front-agent-pm-async.md`（强前台+PM异步）、idea-9/idea-10/idea-11、idea-7（错误账本）

## 背景

M39 已让 WebRTC 通话做到「实时上行 + 流式转写（边说边出字）+ 端点(停顿)触发回复 + rolling transcript 上下文累积」。但触发回复的判据是**停顿（静音边界）**，不是**意图完整**。user 指出更优架构：不靠停顿、靠语义；且把"何时播放"的决策下放到客户端，趁用户还在说就把回复提前算好，藏住 LLM+TTS 延迟，达到极快的对话体感。

## 决策

定下实时通话（WebRTC 通路，**与对讲旧机制分离**）的下一代架构，四个支柱：

### 1. 前台 GLM 一身三职（便宜、快、可反复跑）
- **判意图完整**：GLM 边接流式转写边判"这段意图说完整了没"（语义判定，不是等停顿）。意图判定本就是前台 GLM 的职责，不另加模型。
- **投机生成 + latest-wins 取代**：意图一旦"大概完整"就**立刻**生成回复；用户继续说、意图变了，就**取消上一条还在算的生成、用新的取代**（复用现有 `playbackGeneration` 代际机制）。服务端始终只维护"当前这段话对应的当前最佳回复 + 它的意图代际号"。
- **完整意图交 PM（命门契约）**：需要 PM(opus) 实际动手时，GLM 必须把**一句完整、distill 过的意图**交给 PM，**绝不把半截话碎片一点点丢给 PM**（碎片会让 PM 乱办、出错）。

### 2. 客户端播放闸（turn-taking 下放到 app）
- 服务端把"当前回复 + 意图代际号"流给 app；app **攥着不立即播**。
- app 用**最新鲜的麦克风信号**判断用户是否还在说话：还在说 → hold；用户真正让出话权 → 播放**代际号最新**的那条，旧代际丢弃。
- 何时播放的决策放客户端，省掉一整个 RTT（服务端要慢一拍才知道用户停没停）。

### 3. 通路分离 + 来源打死标签
- 对讲（旧机制，连续/批处理）与通话（WebRTC 实时）**两条路分开**，各走各的。
- 每条来源打死标签（对讲 / 通话 / 普通语音），注入、DB、路由都按标签走，**绝不混**。
- （治旧坑：WebRTC 转写曾与普通语音同 `{source:'voice'}` 无法区分，多次误判。）

### 4. 自我进化评价机制（idea-7 语音化）
- 记录用户提出的问题 + 反馈（user 当场指错 + 事后收集）→ 形成评价机制 → AI 修正自己的意图判定与回复方案，**越用越准**、投机错的越来越少。

## 理由

1. **极快**：投机生成把 LLM+TTS 耗时藏在用户说话时间里；客户端播放闸省掉"能不能播"的 RTT。
2. **成本天然有界**：投机循环里**只有便宜快的 GLM 在跑**（~1-2s、免费 /coding/ 接口）；**贵的 PM(opus) 永不投机、永不吃碎片**，一通话最多在"完整意图"上触发一次；TTS 只合成"将要播的那条"。→ user 的"给 PM 必须完整"契约**同时是正确性要求和省钱杠杆**。
3. **turn-taking 在客户端最准**：app 当下就知道麦克风状态，服务端慢一个 RTT。
4. **学习闭环 = 护城河**：长期把意图判定准确率拉上去，减少白算。

## 已知代价

- **投机白算**：中途意图常被用户后续推翻（"查天气…啊不对，看关羽在干嘛"），被取代的生成花了算力。靠"只 GLM 投机 + 取消纪律 + latest-wins"压住；靠学习闭环长期减少。
- **取消纪律是硬要求**：superseded 的 GLM 生成与 TTS 合成必须**真掐掉**，否则旧音频抢播 / 空跑烧钱。
- **意图完整判定的延迟/成本**：GLM 反复判完整本身有调用开销，需控制频率（不必每个 partial 都判）。
- **协议复杂度**：服务端↔app 要约定代际号 + 意图状态 + 播放闸语义，比"服务端推一条 app 直接播"复杂。
- **native 隔离遗留**（承 M39）：sherpa 流式仍可能 native exit()，彻底稳需子进程化，跟踪项。

## 2026-06-07 设计细化（user 真机马拉松后口述补充，PM 评估采纳）

user 重申并磨利了投机+播放闸架构，补三个关键增量（喂 Phase 2/3 实现）：

1. **回复语音必须分段（命门）**：回复不能是一整块音频，必须切成**独立可单独 攥住/撤销/播放 的小段**。只有分段，才可能"撤掉前面算错的一段"。→ 协议要定 segment 粒度的帧族（segment_id / generation / text / audio chunk），客户端持**段队列**。
2. **撤销规则：未播可撤、在播不撤**：客户端段队列里——还没开始播的段可丢弃/替换；**正在播的那段已提交，不中途砍**（砍了割裂感更糟）。GLM 边理解边发现前面某段错了→请求 app 丢弃**尚未播**的该段；已在播则不动。
3. **段要短**：正在播的段不可撤=该段时间窗内若意图变，用户会听到一句错的；**段越短，"已提交错音频"暴露越小**。倾向短句/短语级细粒度分段。

**PM 评估**：方向正确，=本 ADR 投机+latest-wins+客户端播放闸的同源细化，三点是真增量已采纳。PM 先提三个硬问题，user 驳回前两个（成立），第三个保留：

- **❌→✅ "播触发"不另造信号（user 驳回，对）**：PM 一度提"需 turn-yield 信号"是多余——**播放触发复用现有"判用户停说话→该发"那个节点**即可。两个触发器要分清：**算的触发**=GLM 判意图完整（投机生成）；**播的触发**=现有停说话节点。回复趁说话时已投机算好，用户一停（现有节点触发）立刻播=近零延迟。不重造信号。
- **✅ 撤段连贯=GLM 段注册表+撤回协议（user 给的归属模型，比"整条重生成"干净）**：语音是 GLM 发的，**GLM 给每段打标、自己维护段链**，撤几段由 GLM 决断（只有它知道哪些段依赖那段错的）。**Phase 2 硬接口**：
  - **段注册表（GLM 侧）**：每段 = {segment_id, 文本内容, 依赖上文, 状态: 待播/在播/已播}，GLM 持续维护。
  - **撤回协议（GLM→app）**：GLM 发现错→发"撤回 segment_id≥N"→app 执行：未播的段全丢、**正在播的那段不动**、已播认了。
  - **替换**：撤完 GLM 从纠正点往后重发新段（id 接续），app 排队列等播。
- **⚠️ 保留：投机白算成本 + 撤纪律命门**：只 GLM 投机（便宜）+ TTS 只合成大概率要播的段。工程命门=**撤时 in-flight 的 GLM 生成与 TTS 合成真掐掉、旧段绝不抢在新段前播**。

**前提不变**：先用 Phase 1 shadow 数据证明 GLM 判"意图完整"够准（正在收），准了才敢 Phase 2 让它真驱动——判错就播错段，撤救不过来。STT 用 funasr/Paraformer 系（M39 sherpa-onnx streaming 已是）。

## 下行架构决定（2026-06-07 user 拍板）— 一箭双雕：文件分段播放下行

**根因（赵云深挖 `reports/2026-06-07-webrtc-call-audio-loud-clean-deep-dive`）**：对讲念回响且干净=走 **expo-audio 文件播放**（TTS→MP3→useAudioPlayer + 媒体音量模式，STREAM_MUSIC）；WebRTC 通话下行=播 **libwebrtc 实时 remote track**（走通话流 STREAM_VOICE_CALL，天生小）。两条本质不同管线，对讲响亮配方搬不到 WebRTC track。media 路由发劈=edge-tts +40% 与下行 3x 双重增益 Int16 削波。

**决定（user 拍板"试一下"）**：
- **上行保持 WebRTC**（麦克风→实时 STT，已验证好）。
- **下行改走文件分段播放**：AI 回复当**音频段（文件/chunk）**，用对讲那条 expo-audio 文件播放路播出来（响+干净，媒体流）。
- **一箭双雕**：① 解音量（复用对讲验证过的响亮路，不再跟 WebRTC track 较劲、不双重增益削波）② 解分段（每段=一个文件，app 攥着段队列、撤未播的、播最新的，天然就是撤回协议要的粒度）③ 配合投机=段算好就发、app 择时播=低延迟。**音量根治 = M40 Phase 2 下行架构 = 同一件事。**
- **★硬约束（user）**：**WebRTC 下行能力保留、不删**（flag-gated 当能力/fallback）；只是默认接口不走它、改走文件分段播放路。
- 复用现成：对讲已用 voice_stream/relay 通道播 TTS 文件——通话下行接到同一条播放路，加分段+撤回。
- 不再出 WebRTC-track 音量实验包（那条方向错）。
