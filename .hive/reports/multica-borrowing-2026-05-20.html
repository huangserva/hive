Multica 可借鉴到 Hive 的小颗粒功能二轮调研

最强建议：先做“每个 worker 的 thinking_level 选择”，它和 Hive 已有的 Claude/Codex/OpenCode/Gemini preset 模型最贴合，用户马上能感知，且不需要把 Hive 改成 Multica 式平台。

调研日期：2026-05-20。范围：~/development/multica 最近 100 个 commit，重点 commit #2912/#2865/#2888/#2906/#2904，另看 packages/core、packages/views、packages/ui、server/internal、docs/。Hive 仓库只读分析，未修改 runtime 代码。

结论 1thinking_level 是高 ROI，建议作为下一批功能优先级第一。
结论 2squad 不应照搬，但“复合 assignee picker + leader 解析”这套 UX/state 设计可借。
结论 3runtime/icon polish 可以小步抄：官方 Gemini mark、稳定本机 runtime 行、日志/terminal 视图排序。

按投资回报排序的 Top 8

#1 ⭐⭐⭐ 高 M=1-2天

每个 worker 可选 thinking_level：Claude 用 effort，Codex 用 reasoning effort

Multica 的实现重点不是“统一抽象”，而是保留每个 CLI 的原生命名并按本机 CLI 能力发现。Hive 已经有 command preset、启动参数增强和 worker 创建弹窗，适合做一个轻量版。

src/server/command-preset-defaults.ts:17-62 已有 Claude/Codex preset；src/server/command-preset-store.ts:8-18 可加 nullable thinkingLevel 或单独放 worker launch config；src/server/agent-run-bootstrap.ts:72-95 是启动增强点；web/src/worker/AddWorkerDialogFields.tsx:194-239 是 Add Worker preset UI。
伪代码
type ThinkingLevel = string | null
// create worker payload: thinking_level?: string | null
if (preset.id === "claude" && thinkingLevel) args.push("--effort", thinkingLevel)
if (preset.id === "codex" && thinkingLevel) args.push("-c", `model_reasoning_effort=${thinkingLevel}`)

价值

必须包含项的结论

项目 Multica 怎么做 Hive 能不能 port
thinking_level #2865/#2912 DB 加 nullable thinking_level；服务端验证 provider enum 和本机 model catalog；执行时注入 Claude --effort / Codex reasoning；UI 在 inspector 中按 runtime model catalog 显示 picker。 能，且推荐。Hive 第一版可比 Multica 更小：只做 worker launch config + Add Worker picker + 启动参数注入；动态模型 catalog 可二阶段。
squad assignment #2888 把 assignee 从单一 agent id 扩成 {assignee_type, assignee_id}；UI picker 同列 agents/squads；服务端把 squad 解析到 leader，并留 squad attribution。 只借模式，不借系统。Hive 可在 UI 上借复合 picker，但协议仍保持按 worker name 派单；不建议新增 squad schema。
runtime / desktop / icon #2906/#2904 daemon 停止时保留 local machine placeholder；provider logo 用官方 mark，Gemini 用 Simple Icons CC0；runtime machine 统一健康视觉。 可小步 port。Hive 适合做 runtime 状态条和更准 CLI 图标；不要做 daemon 心跳或远端 runtime 管理。

明确排除

  • 不建议照搬 task lifecycle / timeline / agent presence / skills / squad 作为粗粒度系统，它们服务于 Multica 的多用户平台，不符合 Hive 单人本机轻量定位。
  • 不建议引入 Postgres、Redis、多用户权限、云端 daemon 管理。
  • 不建议做 worker 心跳、超时、卡死检测；这会碰 Hive design spec §3.6 的三态边界。
  • 不建议把 dispatch schema 扩成 cancelled/abandoned 或 group lifecycle；刚修好的 pending/status 路径应保持稳定。

建议落地顺序

顺序 建议 原因
1 thinking_level 后端 schema + 启动参数注入 功能价值最高，和 preset 架构天然贴合。
2 thinking_level Add Worker picker 前端成本小，直接让用户看到能力。
3 错误消息透传补齐 稳定性体验立刻提升，风险低。
4 Gemini 官方图标 + runtime 状态条 小 polish，能减少“我现在跑的是哪个 runtime”的排查成本。
5 复合 picker / role 目标 UI 只有当 Hive 想做批量派单辅助 UI 时再做，避免 scope creep。

最坑的一条

squad assignment 最坑。代码里可借的 picker/state 很漂亮，但如果把它理解成“Hive 也该有 squad”，会马上引出 group schema、leader 转移、dispatch attribution、任务取消、权限和 UI 生命周期，全部偏离当前轻量目标。正确做法是只借 {type,id} 选择器和“服务端最终解析到实际执行者”的思想。
