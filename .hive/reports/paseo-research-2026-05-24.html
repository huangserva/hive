  └─ relay-transport           encrypted remote access

核心数据流：创建并运行 agent

Client createAgent({ provider, cwd, model, mode, prompt, worktree? })
  → Session validates RPC and workspace context
  → AgentManager.create()
  → Provider adapter creates native session
  → Provider emits AgentStreamEvent
  → AgentManager appends timeline row { seq, epoch, timestamp, item }
  → WebSocket broadcasts agent_stream + agent_update
  → App timeline reducer renders chat / tool calls / permission cards

和 HippoTeam 的根本差异

  • Paseo 是 agent runtime 产品：关注多端控制、provider 兼容、移动端体验、worktree/git/PR/relay。
  • HippoTeam 是 PM-driven 协作工作台：关注 orchestrator/worker 协议、tasks/plan/cockpit 文档体系、飞书远控、dispatch lifecycle。
  • Paseo 的 agent 状态机更细：initializing / idle / running / error / closed；HippoTeam spec 固定三态 idle / working / stopped，不能照搬。
  • Paseo 用 JSON 文件持久化：无 schema migration；HippoTeam 已经 SQLite + schema_version，存储层不建议回退。

5. 核心模块读码笔记

模块源码位置读到的模式对 HippoTeam 的意义
Daemon bootstrap packages/server/src/server/bootstrap.ts 单入口装配 HTTP、WS、AgentManager、storage、MCP、relay、schedule、workspace registry。 可借鉴服务组装和显式 dependency wiring；但 HippoTeam 现在 app/store 分层已更轻。
AgentManager agent/agent-manager.ts ManagedAgent discriminated union、subscriber bus、foreground turn waiters、timeline coalescer、attention callback。 timeline/attention 模式值得研究；状态机不能直接 port。
Provider registry agent/provider-registry.ts, provider-manifest.ts provider definition 声明 label、description、mode、icon、risk color、unattended mode；runtime settings 可 override command/env/tools。 强借鉴。HippoTeam 现有 preset/catalog 可升级为能力矩阵，而不是散落 if/else。
Timeline store agent/agent-timeline-store.ts append-only rows，带 seq、epoch、tail/after/before fetch、stale cursor/gap reset。 很适合 Cockpit/TaskLog/Terminal history 的可恢复 WS 模式。
MCP server agent/mcp-server.ts 把 create_agent、send prompt、worktree、schedule、provider list 等暴露给 agent；跨 provider mode 映射。 可借鉴工具 schema 设计；HippoTeam 目前用 team CLI 更简单，MCP 不是短期必须。
Loop service loop-service.ts worker/verifier cycle，支持 verify shell check、verifier prompt、max iterations/time、archive。 和 HippoTeam 自评/review 循环强相关，但要做成显式用户触发，避免违反不做心跳/超时检测的约束。
CLI packages/cli/src/commands/* Docker-style command tree；paseo run 支持 provider/model、thinking、mode、worktree、image、env、label、output-schema。 可借鉴 CLI 输出 schema 和 structured output，但 HippoTeam 的 team CLI 应保持小而稳定。
Skills skills/paseo-*.md handoff, loop, advisor, committee, epic 把操作规程包装成可复用 skill。 和 HippoTeam PM 文档体系高度互补，可转为 protocol/templates，而不是直接安装。

6. 借鉴价值排序

1 Provider catalog 能力矩阵 🟢 强推荐

借鉴点：Paseo 把 provider 的 modes、visual risk tier、unattended mode、voice support、runtime command/env override 统一声明。HippoTeam 已有 command preset 和 thinking_level，可以继续抽象出 supportsThinkingLevel、supportsMcp、riskMode、launchArgs 这类能力。

落地形态：扩展 src/server/command-presets / catalog 层，减少 agent-run-bootstrap 里 provider 特判。工作量 M，风险低。

2 Timeline seq/epoch/gap 模型 🟢 强推荐

借鉴点：Paseo timeline store 的 seq + epoch + cursor 能清楚处理 reconnect、历史分页、stale cursor、gap reset。HippoTeam 目前有 tasks/plan/cockpit WS 和 terminal stream，未来 TaskLog/Feishu log/agent run history 可以统一用这个协议思路。

落地形态：先用于 Cockpit/TaskLog 的事件流，不碰 PTY 数据面。工作量 M，风险中等。

3 Skills 作为 PM playbook 🟢 强推荐

借鉴点：paseo-handoff 的自包含交接模板、paseo-advisor 的 no-edits review、paseo-committee 的对照模型评审、paseo-epic 的 plan-file source of truth，和 HippoTeam PM 体系天然契合。

落地形态：不是直接引入 Paseo skill，而是把这些流程拆成 .hive/templates、ORCHESTRATOR_RULES、Cockpit ActionBar 建议。工作量 S-M，风险低。

4 Loop worker/verifier 显式循环 🟡 看情况

借鉴点：循环不是后台心跳，而是用户/PM 明确启动的"worker → verify → repeat"任务，适合 CI 修复、PR babysit、测试打绿。

风险：HippoTeam design spec 禁止做状态超时/心跳检测；如果做，必须是显式 team loop 或 Cockpit action，且有 max iterations/time。工作量 L。

5 Worktree + archive + PR merge 工作流 🟡 看情况

Paseo 对 worktree、PR、archive 有完整产品化。HippoTeam 当前更像单 workspace PM 控制台，短期只需要"按 dispatch 建 worktree"或"高风险 commit 前生成 review"的小功能，不应整套搬。

6 Relay / mobile app / voice 🟡 到 🔴

技术上很强，但 HippoTeam 已经用飞书桥解决手机入口，移动端原生 app 和 E2E relay 会显著扩大维护面。除非 user 明确要摆脱飞书，否则不推荐近期投入。

7 JSON 文件持久化 🔴 跳过

Paseo 的文件型 store 对用户可读和原子写有好处，但 HippoTeam 已有 SQLite、schema_version、dispatch ledger、agent runs。回退到 JSON 会破坏现有迁移和查询能力。

7. 和 HippoTeam 当前阶段的对照

HippoTeam 当前能力Paseo 对应能力结论
飞书桥：手机远控 + approval + reaction 移动 app + relay + push + voice 目标相同但路线不同。短期继续强化飞书，别开移动 app 维护面。
PM 文档体系：plan/tasks/questions/ideas/baseline/cockpit Skills + epic plan file + loop/advisor/committee Paseo 的 playbook 更成熟，值得转译成 HippoTeam protocol/template。
orchestrator/worker 固定 team 协议 MCP tools 让 agent 创建/控制其他 agent HippoTeam 可保留 CLI 协议；MCP 作为未来可选，不必现在改。
本地 Node runtime + React web Node daemon + Expo app + Electron desktop Paseo 更产品化、更重；HippoTeam 适合继续保持轻量浏览器 UI。
SQLite 持久化 + migration JSON file stores + Zod parse 存储层不建议借鉴；可借鉴原子写和 Zod runtime validation 的局部模式。

8. 推荐后续动作

  1. 短期：把 Paseo provider manifest 的能力矩阵思想吸收到 HippoTeam command preset/catalog，清理启动参数特判。
  2. 短期：把 paseo-handoff、paseo-advisor、paseo-committee 的结构转成 .hive/templates 或 Cockpit ActionBar 建议，不直接引入外部 skill。
  3. 中期：设计 HippoTeam 的 append-only PM event timeline，借鉴 Paseo seq/epoch/cursor，先服务 Cockpit/TaskLog。
  4. 中期：评估显式 team loop，但必须有硬上限，且不做后台心跳/自动超时检测。
  5. 暂不做：移动 app、relay、voice、JSON persistence、大规模 git/PR 产品化。

9. 限制与风险

  • 没有安装运行：按任务要求只读源码，没有 npm install 或启动 Paseo；报告基于 README、docs、package.json、核心源码。
  • AGPL 许可：直接复制代码进 HippoTeam 要注意 AGPL 传染边界。建议只借鉴设计模式，必要时重写。
  • 状态机冲突：Paseo 的 lifecycle 比 HippoTeam 三态更复杂，不能照搬到 worker 状态。
  • 产品方向分叉：Paseo 是通用多端 agent 产品；HippoTeam 是 PM 管理和本地团队协作。过度照搬会稀释 HippoTeam 当前优势。
