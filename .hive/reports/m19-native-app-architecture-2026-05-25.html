M19 HippoTeam 原生 APP 架构设计

user 已拍板：要原生 app，要最佳体验，不因“难”或“和飞书重叠”而降级。 本方案按 epic playbook 锁不可变需求，设计 client/daemon 升级、远程接入层、原生 dashboard/terminal/voice 体验，并拆出阶段闸门。

一句话方案

把 HippoTeam 升级为 first-party client/daemon 产品： 本机 runtime 成为稳定 daemon，公开 versioned HTTP/WS API；Expo / React Native app 是 iOS/Android 第一方客户端； host 配对生成设备凭证；连接优先 LAN direct，失败走端到端加密 relay；app 首屏是 Cockpit + Tasks + Workers dashboard， 深层进入 agent/terminal pane，并最终承载 M14 语音控制多 agent。

Epic 不可变需求

必须满足 原生 iOS/Android app；能安全连接本机 runtime；能看 Cockpit/Tasks/Workers；能进入 agent/terminal； 能从只读走向控制；能与 M14 语音汇合；所有跨设备访问必须有 pairing/auth/authorization。
明确不做 不用 PWA 冒充原生；不默认公开 runtime 到公网；不绕过现有 team protocol/approval；不把 relay 做成可读用户代码/输出的中间人； 不在第一阶段追求 App Store 上架。
成功定义 真机手机可以配对本机 HippoTeam；离开桌面仍能看到实时任务、worker 状态、Cockpit action、agent 输出； 后续能安全下发 team send / approve / stop/restart；语音入口能变成一等交互，而非飞书过渡通道。
变更流程 任何砍掉 remote access、auth、native app、terminal、voice convergence 的提议，都必须回到 user/PM 确认； planner/reviewer 只能指出风险和阶段闸门，不能自行降级需求。

Daemon API 分层

  • Core snapshot API：GET /api/runtime/status, workspaces, workers, terminal-runs, cockpit, tasks。
  • Realtime API：复用并规范 /ws/cockpit/:workspaceId, tasks WS, terminal WS；补 workspace/worker aggregate stream。
  • Control API：mobile-safe wrapper for team protocol：send/report 不给普通 app 用户；app 控制入口映射为 dispatch, approve, stop/restart，并走现有 risk/approval 规则。
  • Pairing/auth API：一次性 pairing code/QR、device registration、token rotation、device revoke、capabilities negotiation。
  • Voice API：先 text command；再 audio upload/local STT；最终 realtime voice session。

远程接入层

这是原生 app 成立的核心硬骨头。HippoTeam 当前绑定 127.0.0.1，手机不能直接访问电脑 loopback。 因此 M19 必须引入 host pairing + direct LAN + relay，不把远程访问当成“以后再说”的 UI 附属项。

Host pairing

runtime 生成一次性二维码：server id、LAN endpoints、relay endpoint、公钥、pairing nonce、过期时间。

手机扫码后完成 challenge/response，runtime 记录 device id + public key + allowed workspace scope。

Direct LAN

runtime 可选择监听 LAN interface；默认仍只 loopback，只有用户启用“Mobile pairing mode”时临时开放。

连接必须 TLS 或 Noise-like app-layer encryption，不能裸 HTTP 传 agent 输出和令牌。

Encrypted relay

relay 只转发密文帧，不读 workspace/agent 内容。参考 paseo relay 思路：daemon outbound 连接 relay，手机连接 relay，双方 E2E channel。

初期可自建轻量 Node/Cloudflare Worker relay；Cloudflare Tunnel 只适合作为 dev/个人部署捷径，不作为唯一架构。

认证与授权

对象 建议 理由
设备凭证 每台设备生成 keypair，private key 存 Expo SecureStore；runtime 只存 public key + device metadata。 避免把长期 bearer token 当唯一根凭证。SecureStore 在 Android 使用 Keystore 加密，在 iOS 使用 Keychain。
会话 token 短期 access token + refresh/rotation；WebSocket handshake 携带 token，连接内绑定 device identity。 手机网络会切换，连接要可恢复；丢手机时可以 revoke device。
权限 scope 到 workspace + capabilities：read dashboard、read terminal、send prompt、approve risk、admin runtime。 原生 app 初期可以只读，后续逐步打开控制，不把所有 UI 等价于 orchestrator token。
高风险动作 复用 team approve / approval ledger；app 可成为 approval surface，但不绕过审批。 保持 HippoTeam 已有安全模型，不因移动端出现新旁路。

App 内容与导航

模块 第一版表现 后续增强
Dashboard 首屏：active workspace、phase、Tasks counters、open questions、high aiActions、worker status。 可 pin 多 workspace；跨 workspace risk summary；push badge。
Cockpit 原生 tabs 显示 Plan / Tasks / Questions / Ideas / Decisions / Research / Baseline / Archive / Reports 数据。 支持 answer/promote/confirm/archive actions；markdown/report viewer。
Tasks 只读任务图 + sprint summary；支持筛选 In progress/Open/Done。 移动端新增/勾选任务，但需 PM policy guard。
Workers worker cards：role/preset/status/pending count/last output。 start/stop/restart、派单、查看 run history。
Agent/Terminal pane agent transcript reader 优先；terminal 作为 advanced view，用 WebView/xterm 或 native terminal renderer。 stdin、resize、binary frame、copy/search、attachments。
Voice 按住说话 → STT → 生成 team command draft → user confirm/send。 realtime voice session、TTS summary、后台通知、语音审批。

与 M14 语音控制收敛

M14a Feishu voice command MVP 是过渡入口，验证“语音能否驱动 PM/team protocol”。M19 原生 app 是终局载体：

  1. Phase 1：移动端语音先生成文字 draft，不直接执行，保持可控。
  2. Phase 2：本地 STT / provider STT 作为 daemon-side capability，app 上传音频或实时流。
  3. Phase 3：语音命令与 Cockpit state 联动，例如“让关羽继续 M19a”、“读一下 high aiActions”。
  4. Phase 4：实时双向语音，支持打断、TTS、长任务 progress narration。

技术选型

  • App：Expo + React Native + Expo Router。理由：paseo 已验证；可 iOS/Android 优先，同栈保留 web；EAS Build/Submit 适合后续分发。
  • State：TanStack Query 或轻量 Zustand + custom WS stores。Dashboard snapshot 走 query，terminal/agent stream 走 event reducer。
  • Secure storage：Expo SecureStore 存 device private key / refresh token；普通 host metadata 用 AsyncStorage。
  • Terminal：第一阶段不做 full terminal input；先 agent transcript。advanced terminal 可用 WebView + xterm.js 复用 web 经验，等确认后再 native 化。
  • Relay：自建最小 E2E relay protocol 优先；Cloudflare Tunnel 可作为个人部署/开发捷径，但不替代产品架构，因为它不能表达 per-device capabilities 和 E2E app protocol。
  • Push：Expo Notifications 起步；生产时处理 APNS/FCM 凭据和 device token revoke。

诚实成本与风险

风险 影响 控制方式
协议未稳定就写 app 移动端反复追 server shape，返工大。 M19a 第一项先做 mobile API audit + versioned schemas；只读 dashboard 验证 shape。
LAN 暴露 runtime 本机代码/agent 输出可能被同网段访问。 默认 loopback；pairing mode 临时开放；强制 auth；设备 scope；可 revoke。
Relay 中间人风险 远程通道涉及敏感项目内容。 relay 只转发密文；daemon/app 做 E2E encryption；不把 token 明文给 relay。
移动网络不稳定 agent stream/terminal 断续。 snapshot + sequence cursor + reconnect resume；terminal input 后置到 M19d。
工作量大 多月级 epic。 阶段闸门：每阶段必须独立可用，不一次性吞全量。

Testing Strategy

  • Server contract：新增 mobile API schemas 的 unit + true HTTP/WS integration tests。
  • Pairing/auth：真 server + SQLite + device registry；覆盖 expired code、revoked device、wrong workspace、capability denied。
  • App：React Native component tests + Expo dev client smoke；关键 flows 用 Maestro/Detox 或 Playwright web build。
  • Relay：daemon/app 双端本地 E2E；assert relay server 只能看到 encrypted frames。
  • Voice/push：真机 tests；Expo Go 不够时用 development build。
