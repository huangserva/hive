M40 Phase 2：WebRTC 上行 + 文件分段播放下行

目标决议：WebRTC 继续负责上行麦克风 / 流式 STT；AI 回复下行默认不再走 WebRTC remote audio track，而改为服务端生成短音频段，经 relay E2E 数据通道推给 App， App 复用对讲的 expo-audio 文件播放路径。WebRTC track 下行保留为 flag-gated fallback，不删除。

2. 推荐新增协议：voice_downlink_segment

旧 voice_stream 是“App 请求一段文本合成 → daemon 回完整音频”。通话下行需要 daemon 主动推、支持分段、撤回、generation latest-wins。为了隔离旧路径，建议新增帧族： 

{
  "type": "voice_downlink_segment",
  "op": "segment_open" | "segment_chunk" | "segment_ready" | "retract" | "turn_close" | "error",
  "call_id": "webrtc-...",
  "turn_id": "turn-...",
  "segment_id": 3,
  "generation": 7,
  "seq": 1,
  "text": "我先查一下关羽的状态。",
  "audio": "base64 chunk",          // segment_chunk
  "mime": "audio/mpeg",
  "format": "mp3",
  "is_final": false,                // 本 turn 最后一段
  "retract_from_segment_id": 3,      // retract
  "reason": "superseded"
}

字段规则

  • call_id：绑定当前 WebRTC call；App 只接受当前 call 的帧，挂断后全部丢弃。
  • turn_id：用户这一轮话权的 ID。M39 final / M40 semantic complete 都应生成 turn。
  • generation：VoiceIntentSession / speculative planner 的代际号。App 永远只播放当前 turn 里 generation 最新的未播段。
  • segment_id：同一 turn 内单调递增。撤回以 segment_id >= N 为粒度。
  • segment_open：告知文本和元信息，可用于 UI “AI 正在准备第 N 段”。
  • segment_chunk：base64 音频切片，按 seq 重组；沿用旧 reassembler 的“按 base64 字符串拼接后一次性解码”纪律。
  • segment_ready：该段音频完整可播。也可由最后一个 chunk done:true 隐式表示；推荐显式 ready，方便无音频错误路径。
  • retract：App 丢弃未播且 segment_id >= retract_from_segment_id 的段；正在播放和已播段不动。

3. App 段队列与播放闸

App 新增 per-call SegmentedDownlinkQueue。它只做队列和播放，不决定“用户是不是说完了”。 播放触发复用现有通话 / 对讲“用户停说话”节点：上行 VAD / streaming endpoint 判用户让出话权后， App 调 queue.releaseLatest(turn_id)。 

state per call:
  currentCallId
  currentTurnId
  releasedTurnIds: Set<turn_id>
  playingSegment: Segment | null
  pendingSegments: Map<turn_id, Map<segment_id, Segment>>
  latestGenerationByTurn: Map<turn_id, number>

on segment_ready:
  if frame.call_id != currentCallId: drop
  if frame.generation < latestGenerationByTurn[turn_id]: drop
  enqueue segment as pending
  if turn_id is released and nothing playing: play next

on user_yield(turn_id):
  mark turn released
  play newest-generation pending segment in segment_id order

on retract(turn_id, from):
  delete pending segments where segment_id >= from
  if playing segment matches: do nothing
  already played: do nothing

播放行为

  • 每段一个完整文件，仍用 expo-audio：player.replace({ uri: data:${mime};base64,... })。
  • 段与段之间串行播放；不叠放。当前段结束事件触发下一段。
  • 播放前设置 allowsRecording:false 获取对讲同款媒体播放响度；若要继续 barge-in，则只保留 VAD 监听所需录音策略，不让它抢占播放器路径。
  • 如果新 generation 到来，未播旧 generation 全丢；已在播段承认，不中断。

4. 服务端段生成与撤回状态机

服务端新增 SegmentedFileDownlinkSession，挂在 WebRTC callee / upstream call 生命周期下：

CallDownlinkSession
  call_id
  workspace_id
  mode: "file_segments" | "webrtc_track"
  currentTurn: {
    turn_id
    generation
    next_segment_id
    segments: [
      { segment_id, text, status: planned|synthesizing|sent|retracted|played? }
    ]
    abortControllers: Map<segment_id, AbortController>
  }

on orch_reply / speculative_reply:
  split text into short segments
  for each segment:
    allocate segment_id
    send segment_open
    synthesize TTS with abort support
    send segment_chunk...
    send segment_ready

on generation superseded:
  abort in-flight GLM/TTS for old generation
  send retract from first affected segment_id
  create new generation and resend from correction point

分段粒度

  • 第一版按短句切：中文标点 。！？；、逗号后长度足够时切；每段建议 6-18 个中文字。
  • 过短片段合并，避免播放器频繁 replace 带来空隙；过长句再按 18-24 字软切。
  • 每段 TTS 独立文件。缺点是段间可能有轻微停顿；收益是撤回窗口短、音量走媒体播放。

5. 投机生成如何接 Phase 1 VoiceIntentSession

Phase 1 的 voice-intent-front.ts 已有结构化 verdict、generation、latest-wins、handoff 闸。 Phase 2d 才让它驱动下行，规则如下：

  • likely_complete/complete + should_speculate_tts：生成 speculative reply 段，但只入 App hold 队列，不立即播。
  • 语义 key 改变导致 generation+1：服务端 abort 旧 GLM/TTS，向 App 发 retract，再发新 generation 段。
  • complete + escalate + confidence>=0.75：只触发一次 PM handoff，且 handoff 内容必须是 distilled_intent，不是 partial 碎片。
  • PM / orchestrator 最终回复回来后，作为更高置信 generation 或新的 turn 段进入同一队列；如果前台已播“我去安排”，最终结果可作为后续段播放。
取消纪律：撤回不仅是 App 丢段，服务端也必须真 abort in-flight GLM 请求和 TTS 合成；旧 generation 的 TTS 完成后必须检查 generation，不得再发送 segment_ready。

6. WebRTC 下行保留开关

新增服务端和移动端一致的下行模式开关：

HIVE_WEBRTC_DOWNLINK_MODE=file_segments | webrtc_track | dual_shadow
EXPO_PUBLIC_WEBRTC_DOWNLINK_MODE=file_segments | webrtc_track | dual_shadow
  • file_segments：默认。不开 WebRTC downlink audio track；AI 下行全走文件分段。
  • webrtc_track：保留旧能力，继续 createWebRtcDownlinkAudio + remote track。
  • dual_shadow：只给诊断包用。WebRTC track 可继续生成但 App 默认静音/不播，比较延迟和日志，不给用户日常包开。

src/cli/hive.ts 当前直接把 downlinkAudio:createWebRtcDownlinkAudio(...) 注入 callee。实现时应改为按 HIVE_WEBRTC_DOWNLINK_MODE 选择：默认不注入 downlinkAudio，而注入新的 segmentedFileDownlink。

7. 分阶段实现路线

阶段 目标 主要文件 验收
2a：通话回复走文件播放 不做分段撤回，先把 WebRTC 通话 AI 回复从 remote track 改到文件播放，验证响度和清晰度。 src/cli/hive.ts、新增 webrtc-file-downlink.ts、relay-connector.ts、mobile-runtime-context.tsx、app/call.tsx 真机通话中 AI 声音接近对讲念回；WebRTC 上行 STT 不回归；flag 切回 webrtc_track 仍可用。
2b：短句分段协议 每条回复切成多个独立音频段，App 队列串行播放。 voice-downlink-segment-protocol.ts、relay-transport.ts、segmented-file-downlink-player.ts 2-3 段回复按序播放，无叠音；段间空隙可接受；单测覆盖乱序 chunk 重组。
2c：撤回协议 支持 retract segment_id>=N，未播丢弃，在播不打断。 同 2b + 服务端段注册表 构造旧 generation 第 2/3 段未播时撤回，App 只播新段；正在播段不中断。
2d：投机生成接入 接 VoiceIntentSession verdict，likely_complete 先算段，用户停说话即播最新 generation。 webrtc-upstream-audio.ts、voice-intent-front.ts、新 downlink session shadow 日志证明 GLM 判定可靠后开启；旧 generation 不抢播；PM handoff 只吃完整 distilled_intent。

8. 测试与风险

测试计划

  • 协议单测：segment 帧 schema、chunk 重组、retract 语义、generation latest-wins。
  • App 队列单测：hold/release、在播不撤、未播撤回、旧 generation 丢弃、播放完成接下一段。
  • 服务端单测：TTS 失败发 error、abort 后不发 ready、mode flag 选择 file vs WebRTC。
  • 真机验收：响度/清晰度、barge-in、用户停说话后首段启动延迟、段间 gap。

主要风险

  • 文件段播放不是 WebRTC remote track，严格意义上不是同一媒体通道；但这是为响度和撤回能力主动取舍。
  • 段太短会有播放器切换 gap；段太长会降低撤回精度。需要真机日志调 6-18 字的默认段长。
  • 如果播放前强制 allowsRecording:false，barge-in 监听可能受影响；2a 要先验“响度”，barge-in 联动单独跟进。
  • relay E2E 数据通道承载音频文件段会增加 data 帧流量；但短段 MP3 通常可控，且比 WebRTC 下行音质问题更可控。
推荐拍板：先派 2a，只做“通话 AI 回复走文件播放路 + flag 保留 WebRTC 下行”，不做投机和撤回。若 2a 真机响度/清晰度过关，再派 2b/2c 做分段与撤回，最后接 Phase 1 VoiceIntentSession 的投机驱动。
