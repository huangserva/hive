Paseo 调研 v2：把它看成“手机 + 语音 + 多 agent runtime”

一句话结论：Paseo 的核心不是一个 PM 看板，也不只是一个 agent UI；它是在把 coding agent 变成可随身操控的本地 runtime，关键突破点是移动端原生体验、双向语音、relay 远控、多 provider catalog、可恢复 timeline，以及可组合的 skills playbook。

Section 1：Paseo 核心亮点（客观视角）

1. 作者想解决的问题

Paseo 文档里的定位很明确：它是一个 self-hostable platform，用来在你自己的机器上运行和编排 coding agents，并通过 mobile / desktop / web / CLI / API 从任何地方驱动它们。官方文档强调 daemon-client 架构、local-first、remote access 是架构一部分、移动端是 React Native 原生客户端而不是 webview。证据见 public-docs/why.md:10-16、docs/architecture.md:1-40。

它要解决的具体痛点可以拆成四个：

  • 人在电脑外也能控制 agent：daemon 跑在 laptop / homelab / server，手机通过 relay 或 direct WebSocket 接入；remote access 不只是网页套壳。
  • agent 工具链碎片化：Claude Code、Codex、Copilot ACP、OpenCode、Pi 都进入同一套 session / timeline / permission / provider mode 抽象。
  • 多 agent 并行需要可恢复事件流：agent 输出不是只显示 terminal 文本，而是规范化为 timeline rows，带 seq、epoch、gap/stale cursor 语义。
  • 手机上打字不足以驱动开发：一等语音能力包括 dictation 和实时 voice conversation，语音不是外部输入法层面的附属功能。

2. 跟同类 coding agent 工具的差异化

3. 语音控制实现细节

语音是 paseo 最值得重读的部分。它由五层组成：

3.1 设备侧双向音频：expo-two-way-audio

packages/expo-two-way-audio 是一个独立 Expo native module，README 说明目标是 real-time conversational apps：采集 mic PCM、播放 PCM、暴露输入/输出音量、支持 iOS mic mode。关键实现不是 JS 录音 API，而是 iOS/Android 原生层。

  • iOS：AudioEngine.swift 使用 AVAudioEngine + AVAudioPlayerNode，会把 AVAudioSession 设置为 .playAndRecord / .voiceChat，打开 speaker 与蓝牙选项（ios/AudioEngine.swift:80-99）。初始化时创建 16kHz mono format（lines 35-42），开启 voice processing（lines 102-111），用 input tap 输出 Int16 PCM（lines 119-158）。播放第一批 PCM 时还会短暂丢弃麦克风输入，让 AEC 适应 speaker playback（lines 185-197）。
  • Android：AudioEngine.kt 设定 16kHz、PCM 16-bit、mono（android/AudioEngine.kt:23-27），音频模式为 MODE_IN_COMMUNICATION，根据外接设备动态路由（lines 82-168）。录音使用 AudioRecord(MediaRecorder.AudioSource.VOICE_COMMUNICATION)，打开 AcousticEchoCanceler 和 NoiseSuppressor（lines 199-229），然后 executor 循环读取 mic buffer 并回调给 JS（lines 236-249）。
  • Expo bridge：iOS/Android module 对外暴露 initialize、playPCMData、toggleRecording、stopPlayback、pausePlayback、mic permission，以及 onMicrophoneData / volume / interruption events。JS 侧 audio-engine.native.ts 再做 permission、playback queue、PCM resampling。

3.2 App 侧语音 runtime

packages/app/src/voice/voice-runtime.ts 把音频流和 daemon session 串起来。它定义状态 disabled / starting / listening / submitting / waiting / playing / stopping（voice-runtime.ts:24-31），并暴露 startVoice、stopVoice、handleCapturePcm、handleAudioOutput、onServerSpeechStateChanged 等接口（lines 159-180）。

  • 采集端：手机端用 16kHz PCM，handleCapturePcm 进入 uploader，转换 base64 后通过 daemon session adapter 发 sendVoiceAudioChunk（voice-runtime.ts:498-515、640-645）。
  • 播放端：server 发来的 audio_output 被按 groupId/chunkIndex 排序，voice runtime 只在 voice mode 且对应 server/agent 活跃时播放；每个 chunk 播放后 ack（voice-runtime.ts:333-390、668-710）。
  • 重连端：updateSessionConnection 在连接恢复时调用 resyncVoiceMode，把 voice mode 状态重新同步到 daemon（voice-runtime.ts:624-637）。
  • 体验端：等待 agent 时有 thinking tone cue，防止手机用户误以为断了（voice-runtime.ts:429-495）。

3.3 Dictation stream：seq / replay / finish

DictationStreamSender 是一个很小但重要的非 React state machine。它维护有序 PCM segment buffer、sendSeq、dictationId、stream readiness。断线时 segment 留在内存；连接恢复后重启 stream 并从 sendSeq=0 重放，finish 前确保所有 segment flush（packages/app/src/dictation/dictation-stream-sender.ts:15-26、83-118、121-188）。这套设计说明 paseo 对移动网络不稳定有明确建模。

3.4 Server 侧 STT / VAD / TTS

Server 侧有两条 speech provider 路径：

  • Local speech：文档默认 STT model 是 parakeet-tdt-0.6b-v3-int8，TTS 是 kokoro-en-v0_19，模型在 daemon startup 下载到 $PASEO_HOME/models/local-speech（public-docs/voice.md:25-52）。源码里 providers/local/runtime.ts 会根据 dictation/voice STT/TTS 的配置加载 Sherpa ONNX、Pocket TTS、Silero VAD。
  • OpenAI speech：providers/openai/runtime.ts 根据配置初始化 OpenAI STT/TTS/dictation。dictation 使用 OpenAIRealtimeTranscriptionSession，连接 wss://api.openai.com/v1/realtime?intent=transcription，发送 session.update，把 input_audio_buffer.append/commit 与 transcript delta/completed 映射为内部 streaming transcription events（realtime-transcription-session.ts:21-62、98-205）。TTS 用 OpenAI audio speech API，默认 tts-1、alloy、pcm（tts.ts:1-61）。

实时 voice conversation 的 turn 由 voice-turn-controller.ts 处理。它组合 VAD provider 与 STT provider，过滤 filler partials，跟踪 capturing utterance、committed segment、final transcript timeout，并在 STT error 时尝试一次 session reconnect（voice-turn-controller.ts:14-42、93-115、196-257）。

3.5 Voice agent 与 speak MCP tool

语音不是直接改主 agent 的普通 prompt。voice-config.ts 会在 system prompt 里插入 <paseo_voice_mode> block，要求 agent 始终用 speak tool 面向用户发声，并把 spoken input 包成 <spoken-input>（voice-config.ts:1-58）。MCP server 里有 voice-only speak tool，注释说明 hidden voice agents 用它通过 daemon-managed TTS 发声（agent/mcp-server.ts 中 voice speak path）。这让 voice mode 成为一套完整交互协议，而非单次输入增强。

4. Mobile-first 架构

Paseo 的 mobile-first 体现在“移动端是原生客户端 + daemon 是本地执行面 + relay 是安全连接面”三层：

  • Expo app：packages/app 同时服务 iOS、Android、web 和 desktop shared UI。SessionContext 负责 daemon client、agent state、notification、voice runtime adapter 和 timeline reducer。voice runtime 注册 session adapter 后，通过同一 daemon client 发 setVoiceMode、sendVoiceAudioChunk、audioPlayed（session-context.tsx voice registration path）。
  • 连接恢复：app store 和 session context 会在 reconnect 后 hydrate workspaces、prefetch provider snapshot，并让 voice runtime resync active voice mode。
  • 后台与可见性：代码里使用 expo-keep-awake 防止 voice session 期间休眠，且引入 push token、OS notification、app visibility 相关模块，说明 mobile client 有持续运行与注意力提醒模型。
  • Relay：官方安全文档把 relay 作为手机最简单连接方式：daemon outbound 到 relay，手机在那里会合，不需要开端口；relay 被设计为 untrusted，端到端加密（public-docs/security.md:18-38）。

5. Relay / E2EE 设计

packages/relay 把加密通道封装成 client/daemon 对称 API。安全文档说明 daemon 生成持久 ECDH keypair 存到 $PASEO_HOME/daemon-keypair.json，QR / pairing link 传 daemon public key，客户端发自己的 public key，双方 Curve25519 ECDH 得出 shared key，后续用 XSalsa20-Poly1305 加密（public-docs/security.md:31-37）。源码 encrypted-channel.ts 里 client 侧发送 e2ee_hello，daemon 侧派生 shared key 并回 e2ee_ready，且有握手重试与 post-hello plaintext guard；crypto.ts 使用 tweetnacl 的 box.before、box.after、24-byte nonce。

6. Provider catalog 设计

provider-manifest.ts 把每个 provider 的 mode、安全颜色、icon、是否 unattended、voice defaults 声明出来。Claude 有 default/auto/acceptEdits/plan/bypass，Codex 有 auto/auto-review/full-access，OpenCode 有 build/full-access/plan；Claude/Codex/OpenCode 还声明 voice default mode/model（provider-manifest.ts:4-31、34-209）。

provider-registry.ts 再把 manifest 与 factory 合并：内建 direct provider 覆盖 Claude、Codex、OpenCode、Pi、Mock，也有 Copilot/Cursor/Generic ACP；custom provider 可以 extend built-ins 或 acp，runtime settings 可覆盖 command/env/disallowedTools（provider-registry.ts:50-129、152-190）。官方 docs 也明确 ACP 是推荐集成方式，direct 适合需要全控制的 provider（docs/providers.md:5-23）。

7. Timeline / seq / epoch / gap 模型

Paseo 的 timeline 是 app 能跨设备、断线、分页恢复的基础。InMemoryAgentTimelineStore 为每个 agent 维护 epoch、rows、nextSeq；append 时分配递增 seq；fetch 支持 tail、after、before；cursor epoch 不匹配返回 reset + staleCursor；cursor seq 落在窗口前返回 reset + gap（agent-timeline-store.ts:17-24、182-240、243-257）。架构文档强调 timeline append-only、epoch 每次 run 更新、默认 fetch page 200，并且 timestamp 由 daemon 作为 canonical source（docs/architecture.md:190-215）。

8. Skills playbook 系统

9. Agent lifecycle / Session 模型

Paseo 的 agent lifecycle 比 HippoTeam 当前 worker 三态更丰富：initializing → idle ⇄ running，失败进入 error，最终 closed（docs/architecture.md:190-207）。这更适合“agent 是可长期存在的会话”模型：一个 agent 可以被 archive、resume、reload、update mode/model；provider persistence handle 记录 Claude resume token、Codex thread ID 等（docs/data-model.md:43-99）。

Session 层则把所有客户端连接都当成订阅者：WebSocket hello 携带 client capabilities，server 后续根据 capability gate 新字段；同一条 WS 同时传 JSON session messages 和 terminal/file binary frames（docs/architecture.md:126-179）。

10. Persistence 模型

Paseo 使用文件型 JSON persistence，不是 SQLite。docs/data-model.md 明说所有 server store 在 $PASEO_HOME，Zod runtime validation，大部分 store 采用 temp file + rename 原子写，没有 schema migration framework，而是靠 optional fields + defaults 做向前兼容（docs/data-model.md:1-5）。目录包括 config、server-id、daemon-keypair、agents、schedules、chat、loops、projects、push-tokens（docs/data-model.md:9-33）。

JSON 文件模型优点SQLite 模型优点
用户可直接读/备份/同步；单 agent 一个 JSON 文件便于 inspect；没有 DB migration 心智；跨平台打包简单。 事务一致性、并发写控制、复杂查询和索引、聚合统计、历史迁移更可控。
适合 local-first 桌面/手机 daemon 早期快速迭代。 适合 HippoTeam 这类已经有 runtime.sqlite、dispatch ledger、run records 的状态系统。

11. packages/highlight：移动端代码渲染小模块

packages/highlight 是一个独立语法高亮包，基于 Lezer parsers，把 JS/TS/TSX/Python/Go/Rust/JSON/CSS/HTML/Java/Elixir/Markdown/YAML 等扩展映射到 parser，输出 HighlightToken[][]，并提供 light/dark color maps。亮点不是算法复杂，而是它避免在移动端直接引入完整 editor/highlight 框架，给 chat/timeline/code diff 提供轻量 tokenization（packages/highlight/src/parsers.ts、highlighter.ts）。

结论

v2 的 framing 应该把 Paseo 放在“agent runtime + mobile control + voice interface”的坐标系里看。它最值得研究的不是某一个 UI 面板，而是把本地 agent daemon、跨设备客户端、端到端语音、provider catalog、可恢复 timeline、和多 agent playbook 合成一个产品面的方式。对 HippoTeam 来说，当前可以先吸收 timeline / catalog / playbook；如果 user 决定推进 mobile + voice 方向，Paseo 的音频桥、speech runtime、voice prompt/speak tool、relay E2EE 是最直接的参考路径。
