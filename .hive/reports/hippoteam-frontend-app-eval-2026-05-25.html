HippoTeam Frontend App 路线评估

目标来自 user 飞书原话：“Paseo 是有 APP 端的，我们是不是可以为 HippoTeam 做一个前端 APP？这样所有任务看起来很方便，也可以有面板。” 核心不是“必须移动端”，而是更方便地看所有任务、worker 和 Cockpit 面板。

推荐

先做 PWA / installable web app，而不是立刻自建原生移动端或桌面壳。 HippoTeam 已经有成熟 web UI、Cockpit 9 tabs、Tasks tab、Reports/Research/Decisions/Baseline/Workers 等面板。 第一阶段的最大增量应是让它“像 app 一样打开”：manifest + icon + standalone display + install CTA + dashboard-first layout。

如果后续 user 明确要“桌面本地 app 自动带 runtime”，再做 Tauri/Electron。若明确要“手机远程看和控制”，优先评估飞书桥是否已满足； 只有飞书无法承载任务面板体验时，才进入 Expo / React Native + relay / tunnel 级别的移动 app。

paseo app 端拆解

paseo 不是单一 app，而是一套 local-first client/daemon 架构：Node daemon 管 agent，客户端通过 WebSocket 连接。 它有 Expo mobile/web app、Electron desktop wrapper、CLI、可选 encrypted relay。这个形态服务的是“跨设备控制同一 daemon”，不是单纯把网页套壳。

Expo app

形态：packages/app 是 Expo / React Native / Expo Router 项目，同时支持 iOS、Android、web export。

能力：projects/sessions/workspace screens、agent list、split panes、terminal webview、voice/dictation、host pairing。

Desktop app

形态：packages/desktop 是 Electron wrapper，依赖 server/cli，能管理 daemon subprocess。

亮点：桌面用户不需要手动打开浏览器和 daemon，app 负责本机 runtime 生命周期。

Relay

形态：packages/relay 是端到端加密远程桥，mobile 可 direct 或 relay 连接 daemon。

亮点：解决手机不在本机网络、daemon behind firewall 的远程访问问题。

paseo 的亮点在于：同一 daemon protocol 被 mobile/web/desktop/CLI 共享；mobile 不是“浏览器打开 localhost”，而是有 host profile、 direct TCP / relay / socket / pipe 连接选择、连接探测、agent directory revalidation、workspace navigation 和 native 权限能力。

HippoTeam 当前状态

HippoTeam 当前是本地 Node runtime + Vite/React web UI，server 绑定 127.0.0.1。Web UI 已经覆盖用户说的“任务看板/面板”大部分核心：

  • Cockpit 9 tabs：Plan / Tasks / Questions / Ideas / Decisions / Research / Baseline / Archive / Reports。
  • Workers 卡片：左侧能看到每个 worker 状态、preset、pending task count。
  • Tasks tab / Todo mini drawer：任务图和 sprint 状态已经实时同步。
  • RuntimeStatusStrip：最近新增 port / pid / cwd / logs / db / version 信息，已缓解“连的是哪个 runtime”。
  • Feishu bridge：M4 已实现移动/远程文本控制、outbound reply、approval cards，是现有 remote control 路线。

因此 app 化的第一步不是重写面板，而是把现有面板变成更容易打开、固定、独立窗口化的 app 体验。

路线对比

路线 实现要什么 工作量 契合度 维护成本 / 风险
PWA installable web 给现有 web 加 manifest、icons、theme/background color、standalone display、install CTA；可选 service worker 只缓存静态壳，不缓存实时 API。 小：0.5-1.5 天可做第一版。 最高：直接复用现有 Cockpit / Task / Worker UI。 仍依赖本机 runtime 已启动；手机访问 127.0.0.1 不成立，除非同机浏览器或另配远程入口。
Tauri / Electron desktop shell 桌面壳加载现有 web，管理/发现本机 runtime，提供 tray、deep link、auto update、日志入口。Tauri 更小，Electron 更成熟且 paseo 已选 Electron。 中：2-5 天原型，稳定分发更久。 中高：适合“本机 app”体验，但对“看任务面板”不如 PWA 直接。 需要新打包链、签名、auto update、跨平台调试；Tauri 引入 Rust/系统 WebView 差异，Electron 体积和安全面更大。
Expo / React Native mobile 新建 RN app、host pairing、auth、push、mobile dashboard、远程 tunnel/relay，可能复用 M14 voice/audio 能力。 大：4-8 周级，取决于远程接入和 App Store 需求。 中：适合长期移动控制，但会与 Feishu 远控和 M14 voice 路线重叠。 双端发布、证书、真机测试、后台限制、推送、远程安全模型；如果只是看任务面板，ROI 过低。

关键架构约束

127.0.0.1 绑定决定了“手机 app”不是 UI 问题

HippoTeam runtime 当前只监听本机 loopback。桌面浏览器/PWA 能访问；同一台机器上的 installable app 也能访问。 但手机上的原生 app 无法访问电脑的 127.0.0.1。要做真正移动远程 app，必须引入至少一种远程访问层： LAN host binding + auth、Cloudflare Tunnel/自托管 relay、或类似 paseo 的 E2E relay。

飞书桥已经覆盖“远程随时控制”的第一层

HippoTeam 已经有 Feishu inbound/outbound/approval cards。若 user 主要想离开电脑后发任务、看结果、审批高风险动作， 飞书桥比自建移动 app 更便宜。自建 mobile 的合理触发条件应是：user 明确需要可视化任务面板、长时间浏览 worker 输出、批量管理任务， 且飞书消息流无法承载。

Dashboard 增量在哪里

“做 app”不应等同于“重做 dashboard”。现有 Cockpit 已经是 PM dashboard；下一步更像产品化入口：

  1. Install shell：PWA manifest + install CTA + icon，让用户从 Dock/桌面/启动台打开 HippoTeam。
  2. Dashboard-first landing：宽屏默认显示 Workers + Tasks + Cockpit summary 的常驻面板，而不是必须点 drawer。
  3. Responsive task board：手机/窄屏保留只读任务摘要与 ActionBar，不强行塞完整 terminal。
  4. Runtime discovery hints：把 RuntimeStatusStrip 的 port/log/db 信息变成 app 内“连接当前本机 runtime”的明确状态。

推荐路线

阶段 做什么 成功标准
M-app-1 PWA：manifest/icons/standalone/install CTA；不引入 service worker API cache，避免实时数据旧化。 用户能从桌面/Dock 启动 HippoTeam，打开后连接当前 4010/5180 runtime，不再像普通网页。
M-app-2 Dashboard layout：把 Cockpit summary、Tasks、Workers 做成首页常驻面板；drawer 仍保留深 drilldown。 用户无需点多个入口就能看到所有任务、待答 Q、AI actions、worker 状态。
M-app-3 Desktop shell spike：若 PWA 仍不够，比较 Tauri/Electron 是否需要托管 runtime lifecycle。 明确是否值得为“自动启动 runtime + tray + native open logs”付出打包维护成本。
M-app-4 Mobile remote app 决策：只有当 Feishu 无法满足可视化面板时，才设计 relay/tunnel + mobile app。 有明确远程 dashboard 用例，且安全/连接模型已先行拍板。

ADR 建议

这次路线已经足够清晰，建议起草 ADR：HippoTeam app 化先采用 PWA-first，不立刻重写为 native mobile。 该 ADR 不否定 M14 voice/mobile 的长期方向，只把“任务看板/面板 app”与“语音/移动原生能力”拆成两个问题。
