M14a Feishu Voice Spike

日期：2026-05-25 · 范围：飞书语音消息 → 下载/转写 spike + 第一刀代码实现 · 不含真实飞书 E2E

一句话结论：飞书 SDK 已具备三块基础能力：audio 消息事件、消息资源下载、speech_to_text 文件识别；本次已把 audio → 下载 → 飞书内置 ASR → 复用现有 inbound 注入链路打通为第一刀实现。但 STT 是否采用飞书内置仍需 user 拍板，因为它需要额外语音识别权限，且飞书官方说明免费版不支持该能力。

1. 三个未知结论

1. audio 事件 payload

可实现

SDK 的 im.message.receive_v1 事件类型暴露 message.message_type: string 和 message.content: string。飞书 audio 消息 content 是 JSON string，关键字段为 file_key，通常还包含 duration。

2. 语音资源下载

SDK 支持

client.im.v1.messageResource.get 支持下载消息里的音频、视频、图片和文件。入参是 message_id、file_key、type，返回 readable stream。

3. STT 转写

需拍板

SDK 有 client.speech_to_text.v1.speech.fileRecognize，适合 60 秒以内音频文件识别。但该能力需要语音识别权限，官方文档标注免费版不支持。MVP 不能默认接外部 ASR。

2. 第一刀实现范围

这次实现只做最小闭环，不改变现有 Feishu bridge 的架构：

  1. 新增 parseAudioContent()：从 Feishu audio content 里解析 file_key 和可选 duration。
  2. FeishuTransport.handleMessageReceive 识别 message_type === "audio"。
  3. audio 分支调用 messageResource.get({ type: "audio" }) 下载语音资源。
  4. 把资源 stream 读成 Buffer，base64 后调用 speech_to_text.v1.speech.fileRecognize。
  5. 拿到 recognition_text 后构造成 sourceType: "voice" 的 inbound event。
  6. 后续完全复用现有路由链：resolveRoute → handleFeishuInbound → recordUserInput → orch stdin。

3. STT 决策点

选项需要什么优点代价 / 风险建议
飞书内置 speech_to_text 在飞书开放平台给应用开通语音识别相关权限；确认租户版本支持该 API。 零新增 npm 依赖；数据留在飞书/租户能力范围内；工程上最短。 免费版不支持；权限和格式支持需真实 E2E 验证；识别质量由飞书决定。 首选 MVP，如果 user 能接受权限/版本约束。
外部 ASR（OpenAI Whisper 等） 新增 provider 配置、密钥、费用预算、数据出境确认。 识别质量和多语言能力可控；不依赖飞书语音识别能力是否开通。 付费、外部服务、隐私和合规边界更重；不应由 worker 默认接入。 待 user 拍板；本次不实现。
先不转写，只提示 unsupported 无。 最安全。 无法验证“语音控制多 agent”的核心价值。 只适合作为权限不可用时的 fallback。

5. 尚未验证 / 后续 E2E

  • 真实 audio event content：SDK 类型和文档指向 file_key，但租户真实 payload 仍需用户发一条语音后抓 log 确认。
  • 音频格式：当前实现按飞书语音常见格式 opus 调 ASR；真实格式是否需要 pcm/wav/mp3 或额外转码，要靠 E2E 返回码确认。
  • 权限：messageResource.get 需要机器人在会话内且有对应消息权限；speech_to_text 需要 AI 语音识别能力。
  • 群聊 audio：现有 text 分支有 @bot gating；audio 消息本身没有文本 mention 前缀。本次按“收到且 chat 已绑定”处理，后续 E2E 需要确认是否会误收群内普通语音。
