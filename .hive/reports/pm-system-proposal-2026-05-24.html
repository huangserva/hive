PM 体系提案Orchestrator 升级为项目主管

作者：Orchestrator (Claude Opus 4.7)  ·  日期：2026-05-24  ·  状态：等 user 决定深度

1.这件事的起源

你 5/23 问了一个问题：
"如果我希望每个 workspace 作为一个 ORCHESTRATOR 是一个项目主管，负责管理所有的项目的文档、进度等，特别是从全局去看，目前的项目进度是否符合我们的详细规划，你觉得如何做合适？"
我当时给了 A/B/C 三档（纯约定 / 加 UI / 加主动 cron），推荐 A 起步。这份 HTML 是 A 的完整设计稿，把当时只丢出来的几条点子展开成可落地方案。

2.当前 hive 的 PM 能力缺口

hive 设计本意是"项目蜂巢"——orchestrator 是蜂后，worker 是工蜂。但当前实现里 orchestrator 的人设更像"任务调度员"：

能力现状缺什么
派单（task dispatch） ✓ 完整（team send / team report / team cancel） 已具备
当前任务可见 ✓ .hive/tasks.md + Topbar drawer 已具备
长期 roadmap ✗ 无 plan.md 概念 不知道"我们在跑什么大目标"
plan vs actual 校准 ✗ orch 只看当下任务 跑偏不会主动发现
决策审计轨迹 ✗ ADR / 决策记录散落各处 3 个月后 "我当时为什么选 B 而不是 A" 无法追溯
文档归档约定 ✗ 研究报告 / handoff / decisions 乱放 新 session 不知道去哪找历史
模板复用 ✗ 每次新 plan / 新 ADR 从零开始写 结构容易飘移，PM 自己也累

飞书桥 5 phase 期间，我作为这个 workspace 的 orch 实际上已经在用"PM 思维"工作——出 HTML 设计稿、写 handoff、维护 tasks.md。但这些都是我主观自觉，runtime 不强制、新 workspace 不知道、其他 orch（如 HippoMind 那个）压根没养成这习惯。把"主观自觉"形式化成"系统强制"，就是这个提案的全部。

3.设计：4 件具体的事

3.1 标准化 .hive/ 目录

.hive/ ├── plan.md ← roadmap：milestones / scope / success criteria（新增，强制） ├── tasks.md ← 当前 sprint GFM checkbox（已有，Topbar drawer 渲染） ├── PROTOCOL.md ← runtime 自动生成的协议手册（已有） ├── handoff.html ← 当前 session 给下一个 session 的交接（约定，已在用） ├── decisions/ ← ADR 风格决策记录（新增，按需） │ ├── 2026-05-21-feishu-plan-b.md │ └── 2026-05-24-pm-system.md ├── research/ ← 调研笔记 / 探索性 doc（新增，按需） │ └── 2026-05-22-hippomind-deploy.md ├── reports/ ← 交付型 HTML 报告（已在用） │ ├── feishu-bridge-plan-2026-05-21.html │ ├── upstream-diff-2026-05-24.html │ └── pm-system-proposal-2026-05-24.html └── templates/ ← 各文档类型的种子模板（新增，可选高级） ├── plan.template.md ├── adr.template.md └── handoff.template.html

3.3 ADR 标准结构（decisions/YYYY-MM-DD-slug.md）

# 决策：飞书桥路线选 Plan B（自建 runtime 桥）

**日期**: 2026-05-21
**状态**: 已采纳
**关联**: plan.md → Phase 0-5

## 背景
user 想用飞书远控 hive。两条路：
- Plan A: 接现成 claude-channel-feishu plugin
- Plan B: 自建 runtime 桥

## 决策
选 Plan B。

## 理由
1. Plan A 绑 Claude Code 私有 channel 协议，只覆盖 claude orch（hive 是多 preset）
2. Plan B 走 runtime 层，4 preset 通吃
3. 不依赖 cc 升级可能 break 的私有 flag

## 已知代价
- 多 1-1.5 天工程量
- 飞书 SDK 接入要自己写（但可借鉴 plugin 的 30 行 WS 封装）

## 结果（后写）
shipped Phase 0-5，accumulated 16 commits / 757 tests，无重大返工。

ADR 不需要长，重点是把"背景 + 选择 + 理由 + 代价"显式落档。3 个月后想起来都不用读代码就能 reload 决策上下文。

3.4 强制行为（orchestrator system prompt 加 PM 段）

改 src/server/hive-team-guidance.ts，在 ORCHESTRATOR_RULES 加一段（中文 + 英文双语，跟现有 feishu / approval rules 平级）：

你是这个 workspace 的项目主管（PM）。除了派单和汇报，你还要：

1. 维护 .hive/plan.md
   - 项目第一次启动时，从 user 对话中归纳一份 plan，写进 plan.md
   - 每完成一个 milestone：marked done + 记录 commit hash
   - 计划要变更：先跟 user 对齐再 Edit

2. 维护 .hive/tasks.md（已有约定）
   - 只放 GFM `- [x]` / `- [ ]` 当前 sprint 任务
   - 长报告挪 .hive/reports/，调研挪 .hive/research/

3. 重要决策落 .hive/decisions/YYYY-MM-DD-slug.md
   - ADR 格式（背景 / 决策 / 理由 / 代价 / 结果）
   - 触发：架构选择 / 不可逆操作 / 多个选项的取舍

4. 调研报告归类
   - 偏交付（给 user 看的）→ .hive/reports/*.html
   - 偏笔记（给未来 orch 看的）→ .hive/research/*.md

5. Plan vs Actual review
   - 每开 session 第一件事：read plan.md + tasks.md Done，自问"实际 vs 计划差距、有没有跑偏"
   - 每完成一个 milestone：更新 plan.md + 写 / 更新 handoff.html
   - 跑偏要提醒 user，不要默默继续

6. 全局视角
   - 派单前问自己：这事属于 plan.md 哪个 milestone？
   - 做完后离整体目标更近还是更远？
   - 有没有未派但应该派的事？等 user 提醒就是失职

4..hive/templates/：从上游 marketplace 借的概念

上游 1.4.0 加了 template marketplace（EN/ZH agent prompt 市场，commits 99d3821..352dc52 共 7 个）。代码不抄（跟我们方向不一致 + 体积巨大），概念可借：

PM 体系里，每个文档类型（plan / adr / handoff / research）都有"该长什么样"的隐式期望。让 orch 每次从零写，**结构必然飘移**。 解法：在 .hive/templates/ 放种子模板，orch 起手 cp templates/adr.template.md decisions/$(date +%Y-%m-%d)-slug.md 即可。 工程量小（4-5 个模板文件 + runtime 在 workspace 创建时种子）、收益高（结构强制一致）、可演进（用户/项目 override templates 后所有新文档自动跟）。

5 个最小有用模板

模板用途触发场景
plan.template.md新项目 roadmap 起手workspace 创建时 runtime 自动种子
adr.template.md决策记录骨架orch 做架构选择时 cp 出新文件
handoff.template.htmlsession 交接 HTML 骨架orch 完成 milestone 时填充
research.template.md调研笔记骨架orch 启动调研任务时
milestone-review.template.mdplan vs actual review 报告orch 跑周期 review 时

Phase B · Plan-vs-Actual UI 面板 ≈ 1-2 天
从 plan.md 解析 milestones → 跟 tasks.md Done 段交叉 → 显示完成度。让 user 一眼看到"项目跑偏没"。

实现要点

  • plan.md 解析器：YAML frontmatter (gray-matter)+ checkbox-style milestone list
  • 新 RuntimeStore 方法：getProjectStatus(workspaceId) 返回 { milestones, progress, currentPhase, plannedVsActualGap }
  • UI 组件：Sidebar workspace 行下方加一个折叠区"项目进度"，显示进度条 + 当前 phase + 最近 commit
  • 新 HTTP endpoint GET /api/workspaces/:id/project-status

前提：Phase A 已稳定运行 1-2 周，约定真有人坚持，再加 UI 增益。

Phase C · 主动 review 闭环 ≈ 3 天
runtime 定时 / session-resume 时触发 orch 跑 plan reconciliation → 发现 drift 自动通知（push notification / 飞书卡片）。

触发时机

  • 每 24 小时整点
  • session resume 后第一次 orch 启动
  • 每 N 次 dispatch 后

action

  • runtime 给 orch PTY 注入 system 消息："请跑 plan reconciliation"
  • orch 自动 read plan.md + tasks.md → 生成 milestone-review.md（用模板）→ 评估 drift
  • 有 drift → 通过 FeishuTransport.sendApprovalCard 风格的卡片推飞书：
    📊 项目进度提醒：HippoTeam 计划的 M3 应于 5/22 完成，已 delay 2 天

前提：Phase A + B 都稳了，且确实需要被动提醒。不然 over-engineering。

6.跟其他 agent PM 流派对比

方案核心机制跟本提案的关系
Claude Code 的 CLAUDE.md 单文件项目说明 + 用户偏好 同源思想，但只 1 个文件，没分层。我们提案是分层版
Cursor .cursor/rules 规则文件目录，IDE 读取作为 system prompt 同源（文件 = 长期记忆），但只 codegen 用，没 PM 概念
Linear / Jira SaaS 任务管理 + roadmap 人类工具。本提案是给 LLM PM 用的本地文件版本
LangGraph state checkpoints JSON state 序列化到 disk 机制相似但不可读；我们的文件人类可读 + git-able
上游 marketplace EN/ZH prompt 模板共享市场 4 节借用了 templates 概念，但不做共享市场（轻量、本地）
