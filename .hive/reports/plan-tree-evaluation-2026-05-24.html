plan-tree 对 HippoTeam PM 体系的价值评估

调研对象：/Users/huangzongning/development/plan-tree；对照基线：HippoTeam PM Phase A 10322f9 + Phase B 588a9c9。

一句话结论：plan-tree 是一个“规划文档知识树治理 skill”，不适合整体搬进 HippoTeam；最值得借鉴的是轻量三件套：.hive/open-questions.md、.hive/ideas/inbox.md + promote 规则、.hive/baseline/ 项目事实基线。
Top 1：开放问题账本 强推荐 · S

把未决问题从 tasks 和报告里抽出来，避免 orch 把“待 user 拍板”误当成可执行任务。

Top 2：想法 inbox + promote 强推荐 · S

给“以后也许做”的需求一个低承诺入口，成熟后再提升为 plan milestone / ADR / research。

Top 3：baseline 事实基线 强推荐 · M

把模块图、运行流、状态边界、测试门禁、风险热点从 prompt 记忆中落到固定文件。

plan-tree 完整 Inventory

目录只有 4 个有效文件：SKILL.md、agents/openai.yaml、references/legacy-migration.md、references/maintenance-patterns.md。它不是应用代码，而是一套跨项目 Markdown 规划树维护规约。

Authority / Discovery Order

  • Discovery：显式用户路径优先；其次 docs/plantree/README.md；再保留已有成熟树；最后才 bootstrap。
  • Governance：项目只有一个 planning entrypoint；新增 plan root 必须注册；baseline 放项目级事实，具体 plan root 只链接。
  • Authority：root README 说明阅读路径和权威顺序；roadmap、status、decision、topic 发生矛盾时要有最近 index / authority map 裁决。
  • 非项目特化设计：目录名、文件角色和 intent 是通用的；项目事实只进入 baseline/topic，不写进 skill 本身。

🔴 跳过（3 条）

跳过项原因替代
整体迁移到 docs/plantree/HippoTeam 已有 `.hive/` 作为 runtime/PM 根；迁移会破坏现有 UI、watcher 和 orch prompt。保留 `.hive/`，只借概念。
每个 workstream 一个 plan rootHippoTeam 是单 user 本地 fork，Phase B 看板围绕单 plan.md；多 root 会增加 UI 和维护复杂度。用 milestone + ADR/research 链接表达子方向。
完整 large-tree governance / indexes当前文档树还没大到需要 authority-map、phase-map、topics index。等真实痛点出现再加，避免 PPT 工程。

推荐的 HippoTeam Phase C 形态

最小可行扩展：不要导入 plan-tree 代码或目录名；在 `.hive/` 下新增 3 个轻量模板目录/文件，再加强 ORCHESTRATOR_RULES。
新增/调整具体内容UI 是否需要
.hive/open-questions.md只放 unresolved user/product/architecture questions；字段：Question / Context / Needed decision / Linked plan item。暂不需要；PlanDrawer 后续可显示 count。
.hive/ideas/inbox.md低承诺想法池；promoted 后标记链接，不自动进入 roadmap。不需要，纯 orch 维护。
.hive/baseline/module-map、runtime-flows、storage-and-state、test-and-release-gates、risk-hotspots，初始可写 Unknown / needs inventory。不需要；用于 session resume 和 agent context。
prompt 规则“想法不等于任务；问题不等于任务；resolved question 必须迁入 decision/plan；active plan 不吸收历史证据”。无。
PlanDrawer 小增强可选：显示 open questions count、ideas count、baseline stale warning。等 Phase B 稳定后再做。

风险与边界

  • 最大风险是文档负担过重。 HippoTeam 的优势是本地轻量、多 agent 实操快；plan-tree 的完整树适合长期大型规划，但不能强制每个小需求走十个文件。
  • 不要替换 `.hive/plan.md`。 Phase B UI 已围绕它构建，plan-tree 的 roadmap 模型只能作为维护规则参考。
  • 不要让 orch 变成文档书记员。 推荐只把“未决问题、低承诺想法、项目事实基线”这三类从当前混杂状态中拆出。
  • baseline 需要事实驱动。 模板中不知道的内容要写 Unknown / needs inventory，不能让 LLM 编造模块图。

最终判断

plan-tree 对 HippoTeam 的价值是“治理原则和轻量模板”，不是“目录架构”。建议进入 Phase C 的只有 4 条强推荐，其中前三条可以不改 UI 先上线：open-questions.md、ideas/inbox.md、baseline/。其余内容等 Phase A/B 在真实使用中出现文档膨胀、决策丢失、resume 失真后再逐步吸收。
