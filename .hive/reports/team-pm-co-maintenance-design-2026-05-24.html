PM 文档团队共维护 5 层架构

目标：把 Cockpit / .hive/ 从 orchestrator 事后补文档，改成 orchestrator 与 workers 在执行路径中共同维护的公共 working memory。

问题定义

paseo 调研事故说明，单靠 ORCHESTRATOR_RULES 让 PM 事后 review worker report 再补文档是不可靠的。worker 产出了三份 HTML 报告，但没有 research note； Cockpit 没有及时出现 Questions / Ideas / Plan / Research 的状态变化，user 只能靠自己看 dashboard 发现 drift。修复方向不是增加更多 reactive audit，而是把 PM 文档维护前移到 dispatch prompt、worker rules、git hook 和 Cockpit aiActions。

5 层设计

层 实现位置 职责 本次状态
Layer 1 · dispatch prompt src/server/agent-stdin-dispatcher.ts + hive-team-guidance.ts 每次 team send 自动注入 PM_DISPATCH_REMINDER，worker 在任务入口就看到 PM 文档要求。 已实施
Layer 2 · worker policy WORKER_RULES / .hive/PROTOCOL.md / CLAUDE.md / AGENTS.md 把 worker 定义为 Cockpit 共同维护者，而不是只交代码或报告的执行器。 已实施
Layer 3 · pre-commit .husky/pre-commit + scripts/pm-governance-precommit.mjs 提交层阻断 staged reports/*.html 但无同日 research/*.md 的治理违规。 已实施
Layer 4 · cockpit snapshot to PTY agent-run-starter.ts / agent-stdin-dispatcher.ts / cockpit parser 启动或 dispatch 时把当前 phase、open question 数、aiActions 摘要注入所有 PTY agent。 下个 dispatch
Layer 5 · Cockpit audit pm-reports-orphan-detector.ts + cockpit-doc.ts 兜底发现 orphan reports，生成 high priority research audit action。 已实施

数据流

1. Orchestrator
调用 team send Alice "调研 X"
2. Runtime
拼装 dispatch payload，追加 PM co-maintenance reminder
3. Worker
完成任务时同步维护 reports / research / plan / decisions / questions
4. Git hook
commit 前检查 staged reports 与 research 是否同日并存
5. Cockpit
chokidar 重新 parse，aiActions 显示剩余 audit

测试策略

层 测试

与现有 L1 的关系

现有 L1 包括 tasks.md dispatch lifecycle auto-sync 与 baseline staleness git-log 检测。 本设计不重复它们：tasks.md 仍由 runtime 自动维护派单状态；baseline stale 仍由 baseline parser 生成 audit。新增层只覆盖 PM 文档共维护责任，尤其是 reports / research、plan milestone、 decisions、open questions 的前置提醒与提交层约束。

Layer 5 的 orphan detector 是兜底，不是主路径。主路径是 Layer 1/2/3 让 worker 在提交前就完成文档维护。

Layer 4 下迭代设计

在 parseCockpit(workspacePath) 基础上增加一个 compact snapshot builder，只取 current phase、 open question 数、recent aiActions、baseline staleHint。agent-run-starter.ts 在 agent 首次 ready 后注入一次；agent-stdin-dispatcher.ts 在 dispatch payload 中追加最新 snapshot。为避免 token 成本，snapshot 控制在 20 行内，只注入摘要，不注入完整文档。

风险：dispatch 时同步 parseCockpit 可能引入 I/O 延迟；下一迭代应评估是否复用 watcher cache。
