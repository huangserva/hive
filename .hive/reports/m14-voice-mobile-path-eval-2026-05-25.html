M14 Voice / Mobile Path Evaluation

三路线横向对比：自建 mobile app / 第三方语音 agent 框架 / 飞书 + voice plugin。目标不是把当前 HippoTeam 锁死在飞书里，而是为“语音控制多 agent 开发”选第一条最高 ROI 的进入路径。

日期: 2026-05-25
关联: plan.md → M14 mobile + voice 扩张方向；idea-1 paseo expo-two-way-audio
结论: 先走 Feishu voice-to-text remote control，把语音最小可用形态定义成“语音 → 文字 → 现有 orchestrator/team send/approval 链路”。实时双向音频与自建 mobile app 作为第二阶段，而不是第一阶段。

Recommendation

推荐路线：飞书 + voice plugin first，保留自建 realtime mobile 的架构出口。

  • 最小可用形态: 手机飞书发语音消息，优先使用飞书自带转写文本；没有转写时 runtime 下载音频并走 ASR；最终把文字注入既有 Feishu inbound prompt，让 orch 继续用 team send / team approve / team feishu reply。
  • 为什么先这样: user 的北极星是“语音控制多 agent 开发”，第一步需要验证“口述任务 → PM 理解 → 多 agent 执行”是否成立；这不要求实时双向音频，也不要求先上架移动 app。
  • 什么时候升级: 如果用户开始要求连续对话、打断、边听边说、低延迟播报、离开飞书的独立体验，再评估 LiveKit/OpenAI Realtime 或自建 expo-two-way-audio 路线。

Three Routes At A Glance

路线最小形态工作量架构契合主要风险建议
1. 自建 mobile app React Native / Expo 或原生 iOS+Android；用 expo-two-way-audio / AVAudioEngine / AudioRecord 做实时 PCM + AEC；接 OpenAI Realtime 或 STT→LLM→TTS。 4-8 周 MVP 长期战略强；短期要新建 app、认证、设备测试、分发、push、远程连接。 移动音频栈、AEC、权限、后台、App Store、远程 runtime discovery、安全认证。 不做第一步；作为 M14 Phase 2/3 的战略路线保留。
2. 第三方语音 agent 框架 LiveKit / Pipecat / Vapi 等。框架负责 WebRTC、turn detection、STT/TTS/LLM pipeline、telephony 或 mobile frontend。 2-5 周集成 spike 对实时语音强；但会引入第二套 agent/session/room/job 概念，需要谨慎映射到 Hive orchestrator + PTY。 供应商复杂度、云成本、双 runtime、权限模型重叠；Vapi 类托管服务还涉及付费和数据策略。 若要实时双向音频，优先评估 LiveKit；Pipecat 次选；Vapi 需 user 拍板预算/数据。
3. 飞书 + voice plugin 飞书移动端语音消息/语音+文字 → runtime inbound handler → 现有 PM/orch/team protocol；可选 TTS 回语音泡泡。 2-5 天 MVP 最强：M4 已有 WS inbound/outbound、chat binding、approval card、mobile remote-control 心智。 飞书 bot 事件里的 voice transcript/音频下载能力需实测；不是实时 conversation；依赖飞书客户端。 先做。用最少工程验证语音控制 PM + 多 agent 是否真有价值。

Route 1: 自建 Mobile App

What It Needs

  • 移动端：Expo/React Native 或 iOS/Android 原生。
  • 音频：paseo 借鉴的 @getpaseo/expo-two-way-audio / Speechmatics 模块，16kHz PCM、单声道、AEC、输入/输出 volume。
  • 会话：本机 runtime 暴露 remote auth、workspace picker、Cockpit subset、push/notification。
  • 语音模型：OpenAI Realtime（WebRTC/SIP/WebSocket）或 STT→LLM→TTS pipeline。

Fit With HippoTeam

  • 长期 fit 高：真正 mobile-first，能承载 voice dashboard、notifications、agent timeline。
  • 短期 fit 低：HippoTeam 当前是 local runtime + browser UI + Feishu remote bridge；自建 app 会引入 auth、networking、distribution。
  • PTY 模型仍可复用：mobile 不直接跑 agent，只发命令到 runtime/orch。

Risk

  • paseo 的 audio 模块必须真机测试，README 明确模拟器不能验证 AEC / noise reduction / iOS mic modes。
  • 实时双向音频会把问题从“PM command UX”扩大到 media infrastructure。
  • 如果第一步就做 app，很可能在验证产品价值前先卡在移动工程。

Route 2: Third-party Voice Agent Framework

候选成熟度 / License适合点不适合点
LiveKit Agents Apache-2.0；官方文档强调 open-source ecosystem、WebRTC/mobile frontend、telephony、turn detection、interruptions、agent server。 如果 M14 目标升级为“实时双向语音 agent”，LiveKit 是最完整的基础设施候选。 会引入 LiveKit room/agent server/job 模型；HippoTeam 需要把它降维成 orch input/output，而不是重写多 agent runtime。
Pipecat BSD-2-Clause；GitHub 显示 12.5k stars、2026-05-15 latest release；Python voice/multimodal pipeline。 适合 server-side pipeline、可替换 transport/providers、可自托管。 Python stack 与当前 TS runtime 不一致；移动前端/Feishu integration 仍要自己接。
Vapi 托管平台；公开价显示 voice hosting $0.05/min，模型成本另计，ZDR/HIPAA 是付费 add-on。 最快做 phone/web voice prototype，运维最少。 付费、供应商绑定、数据策略需要 user 拍板；不符合 local-first 默认。

Route 3: Feishu + Voice Plugin

Minimum Viable Shape

  1. 用户在飞书移动端向绑定 chat 发送语音/语音+文字。
  2. Feishu transport 收到 message event：如果 payload 已有文本/转写，直接取；否则下载 audio resource，调用 ASR（OpenAI transcription 或可插拔 provider）。
  3. 把 transcript 组装成现有 [来自飞书 chat=...] prompt，调用 recordUserInput 注入 orch。
  4. orch 照旧使用 PM 规则：需要 worker 时 team send，高风险时 team approve，回复时 team feishu reply。
  5. Phase 2 可选：team feishu reply --voice 或 config 开关，把短回复 TTS 成 Feishu audio bubble。

Reality Check

飞书路线不是 realtime voice conversation；它是“移动端语音指令入口”。这恰好匹配 M14 的第一步，因为“语音控制多 agent 开发”最先需要解决的是口述任务、PM 分派、审批、回执，而不是低延迟连续对话。

Recommended M14 Slices

  1. M14a Feishu voice command MVP: audio/text inbound parser spike、download resource、ASR provider config、transcript injection、tests、one real Feishu E2E. Success = 手机语音说“派关羽跑测试并回我摘要”，Cockpit/tasks/Feishu 都闭环。
  2. M14b Voice reply / readout: optional TTS outbound audio bubble，短状态播报；保留 text fallback。
  3. M14c Realtime evaluation gate: 如果用户开始频繁用语音且反馈“想像通话一样连续说”，再比较 LiveKit vs OpenAI Realtime vs self-built expo-two-way-audio。
  4. M14d Native/mobile app: 只有当 Feishu UX 明确限制产品形态时启动。自建 app 不替代 runtime，只作为 remote client。
