Feishu × Hive 桥接实现计划Plan B · 自建 runtime 桥

作者：Orchestrator (Claude Opus 4.7)  ·  日期：2026-05-21  ·  状态：等 user review

1. 一句话定位

在 hive runtime 层加一个 Feishu Transport，把飞书消息当作"web UI 上的一次派单"灌进任意 workspace 的 orchestrator PTY，让 user 在手机上用飞书远控 hive 而不依赖任何 Claude Code 私有 API、四个 preset 都吃得到。

2. 目标 / 非目标

目标（MVP）

  • ✓ 手机飞书发一句话 → hive workspace 的 orch 看到 → 派 worker → 干完回飞书
  • ✓ 飞书来的消息 自动出现在 hive 的 task log 和对话历史（跟 UI 派单完全等价）
  • ✓ 4 个 preset（claude / codex / opencode / gemini）都支持
  • ✓ 飞书消息发送时 hive runtime 离线也能恢复（飞书 WS 自带 ack）
  • ✓ 1 个飞书 bot 支持绑多个 workspace（按 chat_id 路由）

非目标（不做）

  • ✗ 飞书 Allow/Deny 权限卡片（hive 默认 YOLO，用不上）
  • ✗ 文件 / 图片 / 音频 / 表情上传（MVP 只做文本）
  • ✗ 飞书查 hive task 图状态（Phase 2 再加）
  • ✗ 多 user 隔离（同一群里所有人共享 orch）
  • ✗ WebHook 模式（只用 WS 长轮询，公网 IP 都不需要）

4. 分阶段实现

Phase 0 — 凭证 & 配置存储 ≈ 3 小时
飞书 app 创建、凭证落地、workspace 绑定 chat_id 的 schema 准备好。

飞书侧（手动一次性）

  1. open.feishu.cn 建自建应用 → 开启「机器人」能力
  2. 权限：im:message、im:message:send_as_bot、im:resource、im:chat
  3. 事件订阅模式选 WebSocket（不需要公网回调 URL）
  4. 订阅 im.message.receive_v1
  5. 记录 App ID + App Secret

代码改动

文件动作说明
src/server/sqlite-schema-v21.ts NEW 新表 feishu_bindings: id / workspace_id / chat_id / chat_name / enabled / created_at
src/server/feishu-credentials.ts NEW 读 ~/.config/hive/feishu.json，缺失时返回 null（runtime 不崩，只是不启 transport）
src/server/workspace-store-mutations.ts MOD 加 bindFeishuChat(workspaceId, chatId) / unbindFeishuChat / listFeishuBindings

验证

  • 启动 runtime，tail -f ~/.config/hive/logs/runtime-4010.log 看到 "feishu credentials not configured, transport disabled"（正常的降级日志）
  • 填好 feishu.json 重启，日志变 "feishu credentials loaded, app_id=cli_xxx"
Phase 1 — Inbound: 飞书 → orch ≈ 6 小时
飞书发一句话能注入到指定 workspace 的 orch PTY stdin。这是 e2e 跑通的最小子集。

降级 / 错误处理

  • chat_id 没绑 workspace → bot 回 "此聊天未绑定 hive workspace，请管理员在 hive UI 绑定"
  • workspace 的 orch 离线 → bot 回 "Orchestrator 当前未运行（点 Restart 按钮）"
  • 飞书 WS 断线 → SDK 自带重连，runtime 不死

验证

  1. 在 hive UI 把当前 workspace 绑到测试群 chat_id
  2. 飞书群里 @bot 你好
  3. orch PTY 看到注入文本 → claude/codex 正常理解并回复（但暂时只能在 PTY 里看到回复，飞书还看不到——Phase 2 处理）

视觉要点

  • 用户消息（右侧蓝灰泡）：跟普通飞书完全一致，就是平时怎么跟人聊天就怎么发
  • Bot 回复（左侧白泡）：纯文本，最多带 emoji
  • 进行中卡片（中部）：MVP 不做。Phase 5+ 才上，让 user 看到 "正在跑" 的反馈，免得以为 bot 没收到
  • 长输出：> 30KB 切片发；> 200 行的代码或 log 转 markdown 卡片或附件文件
  • 错误反馈：用红色 ❌ + 简短文字，"Orchestrator 未运行，请在网页点 Restart"

MVP 不做的高级形态

  • Markdown 渲染（飞书富文本卡片）—— 第一版纯文本
  • 点击 "查看完整 worker 输出" 跳 hive web UI（需要 hive 在线 + tunnel）
  • 飞书侧 / 命令快捷栏：/status、/stop <worker>、/restart-orch
  • 语音输入（飞书自带，让 hive bot 把音频转文本派单 —— 你出去开车时用）

8. 风险与缓解

风险等级缓解
飞书 SDK @larksuiteoapi/node-sdk 跟 Node 22 ESM 兼容性 中 Phase 0 半小时 spike，import 失败就上 dynamic import 包一层
飞书 WS 长轮询断线频繁 低 SDK 自带重连。runtime 加 reconnect 计数器，> 10 次报警
飞书消息频率限制（200 msg/min/app） 低 MVP 1 个 user 用，不会触发。后期加 outbox 节流
orch system prompt 没遵循"用 team feishu reply" 中 注入文本里强调一次，再不行就用 stop sequence + 包一层"如果你写了 markdown 而没调 team feishu reply，我会自动转发"。Phase 4 测
飞书消息含 @ 多人时解析复杂 低 MVP 只处理 @bot；其他 @ 文本剥掉
跨 workspace 路由错误（消息发到错的 orch） 中 RouteResolver 加 audit log + 注入文本里始终带 sender + chat_id 上下文，让 orch 可以反驳"这不是我的群"
