HippoTeam M17 design · 2026-05-25

paseo skills playbook 体系借鉴设计

结论：不要复制 paseo 的 runtime；把 5 个 playbook 转译成 HippoTeam 的 PM 文档、模板、ORCHESTRATOR_RULES 和 Cockpit ActionBar 建议。优先落地 handoff，因为它最直接补齐 worker rescue、跨 session 续接和 dispatch 质量问题。

1. Handoff最高 ROI，低实现风险
2. Loop把反复验证变成有边界流程
3. Advisor只读第二意见，减少盲点
4. Committee卡住时做双模型根因分析
5. Epic大项目的不可变需求与阶段门

现有上下文

参考源码位于 ~/development/paseo/skills/。本次读了 paseo-handoff、paseo-advisor、paseo-committee、paseo-epic、paseo-loop，并参考 paseo 基础 skill 与已废弃的 paseo-orchestrate。

HippoTeam 已有 .hive/plan.md、tasks.md、open-questions.md、ideas/inbox.md、decisions/、research/、reports/、baseline/、Cockpit 与 ActionBar。缺口不是“没有 PM 状态”，而是缺少可复用的协作操作手册。

总设计

设计裁决：先把 playbook 做成 PM 层能力：模板负责结构化 brief，ORCHESTRATOR_RULES 负责触发条件，Cockpit ActionBar 负责把当前状态里的“该用某个 playbook”暴露出来。运行时自动化放到后续，而不是 M17 第一刀。
层 建议落点 理由
模板 .hive/templates/playbook-*.template.md 沿用现有 PM 模板机制；先解决 brief 缺字段、上下文丢失、验收不清的问题。
规则 src/server/hive-team-guidance.ts 的 ORCHESTRATOR_RULES 新增 “Playbooks” 小节 把“什么时候必须用 playbook”写进 orchestrator 协议，避免靠临场记忆。
Cockpit cockpit-doc.ts 的 aiActions 增加 playbook 类型建议 ActionBar 已是 PM 主动行为入口，适合提醒“准备 handoff brief / 发起 advisor / 启动 bounded loop”。
后续 runtime 必要时再加 team playbook prepare/run advisor、committee、loop 会消耗 worker 或执行命令，先不要自动化到不可控。

逐个 playbook 转译

1. paseo-handoff → HippoTeam handoff brief

优先级 #1 低实现风险 直接改善 worker rescue

paseo 原版本质：把当前任务完整移交给另一个 agent，确保对方不靠聊天上下文猜测。

关键步骤：识别 provider/worktree/task → 整理自包含 handoff prompt → 包含 task、context、files、current state、tried、decisions、acceptance、constraints → 启动接手 agent。

HippoTeam 缺口：现在有 session 级 handoff.html 和 worker team report，但没有任务级移交 brief。关羽 stuck、dispatch 被 rescue、worker 重启时仍依赖 orch 手写长 prompt。

Templates

  • 新增 playbook-handoff.template.md
  • 字段：任务、dispatch id、当前状态、相关文件、已尝试、决策、验收标准、约束、PM 文档影响
  • 输出位置可先用 .hive/reports/ 或 .hive/handoff.html 附录，后续再加 .hive/playbooks/

Rules

  • worker rescue、跨 worker 转派、同任务重启、长任务中断时必须先写 handoff brief
  • 不得改变任务语义：调研任务不能在 handoff 里变成实现任务
  • brief 必须列未验证假设与剩余风险

ActionBar

  • 当 tasks/dispatch 状态显示 stuck、canceled、reassigned 时提示“准备 handoff brief”
  • 当活跃 worker 长时间无 report 且有未完成验收时提示 handoff
  • 第一版可以只基于 tasks.md 文案和 dispatch 状态做保守提醒

2. paseo-loop → bounded worker/verifier loop

优先级 #2 需要边界保护

paseo 原版本质：让 worker 和 verifier 反复迭代，直到明确 verifier 通过或达到停止条件。

关键步骤：明确目标 → 写 worker prompt → 选择 shell check 或 verifier prompt → 设置 max iterations/time → 归档每轮日志 → stop。

HippoTeam 缺口：现在测试失败、E2E audit、外部状态等待都靠 orch 手动重复派单和复跑命令；没有统一的 loop 记录、验收条件和停止线。

Templates

  • 新增 playbook-loop.template.md
  • 字段：目标、worker prompt、verify command、verifier criteria、max iterations、max time、每轮记录、退出原因
  • loop 日志后续可归档到 .hive/archive/YYYY-MM/

Rules

  • 任何 loop 必须有 max iterations 或 max time
  • verifier 只判断事实和证据，不负责给修复方案
  • 失败三轮以上必须升级为 advisor 或 committee，而不是无限重试

ActionBar

  • 当最近 report 显示同一验证命令反复失败时建议 bounded loop
  • 当 tasks 中出现“等待外部检查/PR/check”时建议 babysit loop
  • 初版只建议“准备 loop brief”，不自动执行命令

3. paseo-advisor → read-only second opinion

优先级 #3 适合决策前

paseo 原版本质：问一个只读 advisor 做第二意见，自己保留最终判断。

关键步骤：挑 provider/视角 → 写 sharp question → 给已考虑/已排除方案和相关文件 → 要求 recommendation with reasoning → 综合 advisor verdict 与自己的建议。

HippoTeam 缺口：现在可以临时派 tester/reviewer，但没有明确“只读、单问题、不得改代码”的 advisor 模式；重大 ADR 草稿也缺少二次质询。

Templates

  • 新增 playbook-advisor.template.md
  • 字段：问题、上下文、候选方案、已排除方案、相关文件、需要回答的 verdict、no-edits 约束
  • 可在 ADR draft 里引用 advisor verdict

Rules

  • 不可逆架构选择、付费/外部凭证、多方案取舍前建议 advisor
  • advisor 不写代码、不改 PM 文档，只交意见
  • orch 必须给出采纳/不采纳理由

ActionBar

  • decision draft 存在且未确认时提示“请 advisor 复核”
  • high priority open question 涉及多选项时提示 advisor
  • 需要扩展 AIActionType 为 playbook

4. paseo-committee → two-advisor stuck analysis

优先级 #4 消耗更高

paseo 原版本质：卡住或高风险规划时，让两个高推理 agent 从不同模型家族做根因分析、计划质询和 diff review。

关键步骤：问题级 prompt → 并行双 advisor → challenge assumptions → 合并计划 → 另行实现 → committee review diff → 需要时循环。

HippoTeam 缺口：AGENTS.md 要求 milestone 前自评，但 Hive worker 边界禁止 nested subagents；实际做法应由 orchestrator 派多个 worker，却缺标准 committee brief 和 consensus 记录。

Templates

  • 新增 playbook-committee.template.md
  • 字段：问题、症状、约束、尝试、假设、成员 A/B verdict、共识、分歧、用户需拍板项
  • 可把最终 consensus 附到 ADR 或 milestone review

Rules

  • 同一 bug 多轮失败、设计分歧严重、风险热点改动时启用
  • committee 成员只读，不直接实现
  • 分歧不能被 orch 硬猜时挂 open question

ActionBar

  • 同任务多次 canceled/retry 或 blocked 时提示 committee
  • risk-hotspots stale 且计划涉及 server core 时提示 committee
  • 需要后续从 dispatch history 提取失败计数

5. paseo-epic → large milestone ceremony

优先级 #5 与 plan.md 重叠，需要谨慎

paseo 原版本质：大项目流程：研究、意图捕获、不可变需求、计划、对抗审查、分阶段实施、交付。

关键步骤：建 plan 文件 → capture intent → grill user → planner/reviewer 对抗 → 确认 → phase loop → delivery commit。

HippoTeam 缺口：已有 global plan.md 和 tasks.md，但缺“不可变需求”块、phase gates、planner/reviewer 结构化审查。M13/M16 这类多层架构已经需要 epic 级 ceremony。

Templates

  • 新增 playbook-epic.template.md 或 epic-plan.template.md
  • 建议后续 active docs 放 .hive/epics/YYYY-MM-DD-slug.md
  • 字段：目标、不可变需求、约束、阶段、验证门、交付条件、PM 文档影响

Rules

  • 跨多 worker、多天、多模块或有用户体验承诺时必须建 epic
  • requirements 只由 user/orch 修改，reviewer 只能挑战 HOW
  • 每个 phase 完成要更新 plan/tasks 和验证证据

ActionBar

  • milestone 有多条并行任务但无 epic plan 时提示建 epic
  • phase gate 未验证时提示“补 epic gate”
  • 需要新增 pm-epics-doc.ts 后才能做准

优先级表

Rank Playbook 当前 PM 增益 实现切口 为什么不是更早/更晚
1 Handoff 最高：直接减少 rescue 和跨 worker 续接的信息损耗。 模板 + ORCHESTRATOR_RULES + ActionBar stuck/reassign 提醒。 与现有 plan/tasks 重叠少，且马上能改善当前团队操作。
2 Loop 高：把重复修测试、E2E audit、外部等待变成有边界流程。 模板先行；后续再实现 runtime loop。 价值高，但自动执行风险比 handoff 大，必须先约束 stop 条件。
3 Advisor 中高：提高 ADR 和高风险选择质量。 只读 advisor brief + decision draft ActionBar。 容易做，但不是每天都会触发；排在 handoff/loop 后。
4 Committee 中：适合真正卡住或高风险的问题。 committee brief + blocked/retry 检测。 消耗两个高推理 worker，触发要保守。
5 Epic 中：补大项目 ceremony，但当前已有 plan.md 基础。 epic template + optional .hive/epics parser。 最系统但也最容易和 plan.md 重叠，适合在前 4 个 playbook 稳定后做。

建议的 M17 实施拆分

  1. M17-A Handoff first：新增 handoff playbook template，ORCHESTRATOR_RULES 加 reassign/rescue/stuck 触发条件，Cockpit ActionBar 加保守 handoff 建议。
  2. M17-B Loop brief：新增 loop template 和 bounded loop 规则，只生成 brief，不自动运行。
  3. M17-C Advisor + Committee：新增 read-only advisory 模板与 decision/blocked 触发建议。
  4. M17-D Epic：评估是否引入 .hive/epics/；如果引入，再加 parser 和 Cockpit 展示。

第一刀推荐 handoff：它最符合当前 HippoTeam 的真实痛点，能在不改 runtime 的前提下立刻提升 dispatch 质量，并为后续 advisor/committee/loop 的 brief 格式打样。
