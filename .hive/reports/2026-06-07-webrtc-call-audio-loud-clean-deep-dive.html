WebRTC 通话音量小 / media 路由发劈深挖

结论：日常对讲念回之所以“响且清”，不是因为同一套 WebRTC 下行调得好，而是因为它走的是 expo-audio 的文件播放路径：服务端先生成 MP3/M4A，App 以 data URI 交给媒体播放器播放，并在播放前把 allowsRecording:false。WebRTC 通话下行则是 react-native-webrtc 远端 audio track，经 WebRTC JavaAudioDeviceModule / Android 实时通话音频设备播放。两者不是同一个 Android 音频流，也不是同一个解码 / 缓冲 / 音量控制栈。

2. WebRTC 通话为什么小

当前默认通话路径会进入 react-native-incall-manager： packages/mobile/src/lib/webrtc-incall-manager.ts:47-63 调 manager.start({ auto:false, media:"audio" }) 和 setForceSpeakerphoneOn(true)。其 Android 源码把 defaultAudioMode 设为 AudioManager.MODE_IN_COMMUNICATION，start 时 audioManager.setMode(defaultAudioMode)，并以 AudioManager.STREAM_VOICE_CALL 请求音频焦点。

这和用户体感吻合：Android 的通信 / 通话音频模式通常走“通话音量”，而媒体播放器走“媒体音量”。 100ms 的 RN 音频模式文档也把 MODE_IN_COMMUNICATION 对应 in-call volume， MODE_NORMAL 对应 media volume。也就是说，小声不是单纯 TTS 太小，而是 WebRTC 下行被放进了实时通话音频路线。

3. media 路由为什么会“发劈 / 有杂音”

最可能的组合根因：media 路由确实避开了 InCallManager 的低音量路线， 但 WebRTC 远端 track 仍由 JavaAudioDeviceModule 播实时 PCM，不会变成 expo-audio 的媒体文件播放器；同时服务端已有 edge-tts +40% 与 HIVE_WEBRTC_DOWNLINK_GAIN=3.0 两层增益。原来在小声通话流里不明显的峰值削波， 在 media 路由变响后会更容易被听成“发劈”。

4. media flag 是否生效：判定方法

当前最直接的判定不是靠主观听感，而是看日志和构建 extra：

  • 构建时：EXPO_PUBLIC_WEBRTC_AUDIO_ROUTE=media 或 WEBRTC_AUDIO_ROUTE=media 必须进入 app.config.ts 的 extra.webRtcAudioRoute。
  • 运行时：mobile-runtime-context.tsx:238-241 从 Constants.expoConfig.extra 取值；resolveWebRtcAudioRoute 只接受小写 media，MEDIA 会回落 incall。
  • 真机日志：看到 [WEBRTCDBG] test_call_audio_route_media 才能证明这次通话跳过了 InCallManager。若只看到 test_call_started/connected，没看到该行，就不能认定 media 生效。

建议下一包只补一行无行为日志：在 start call 时打印 webRtcAudioRouteExtra 和最终 resolvedAudioRoute。这比继续猜“flag 是否烤进包”可靠。

5. 能否把对讲那条响亮路线搬到 WebRTC？

不能直接搬：两条管线本质不同

对讲念回播放的是完整音频文件；WebRTC 通话播放的是 remote audio track。react-native-webrtc 的 _setVolume 只是 WebRTC AudioTrack.setVolume(0..10) 的 gain， WebRTC 官方 Android AudioTrack API 也只暴露“给底层 MediaSource 设置 volume gain”， 不负责把输出改成 Expo/Media3 文件播放路线。当前 RN WebRTC JS 层也没有“把远端 track 强制绑定到 Android STREAM_MUSIC + Media3 解码器”的稳定接口。

可行迁移方案：混合下行

真正能复用“对讲响亮干净配方”的方案，是保留 WebRTC 上行用于实时麦克风 / 流式 STT， 但把 AI 下行 TTS 改回现有 voice_stream / talkback 文件播放：daemon 仍生成 TTS， 不再通过 WebRTC remote audio track 播，而是经既有 relay voice_stream 发给 App，再用 expo-audio 的 player.replace(data URI) 播。这样可以直接吃到同一条响亮干净路线。

代价是它不再是“全双工媒体都在 WebRTC track 内”的纯方案；barge-in / 回声仍需要复用现有 VAD 与回声门控。但从用户目标看，先让 AI 声音响且清，比坚持 WebRTC 下行纯度更重要。

6. 推荐下一步

  1. 先补诊断日志，不再盲出实验包：服务端每次下行 TTS 打 edge_volume、downlink_gain、gain 前后 RMS、peak、clipped sample ratio； App 打 extra.webRtcAudioRoute、最终 route、是否看到 test_call_audio_route_media、_setVolume applied/unsupported。
  2. 用日志复验 media 发劈：如果 clip_ratio > 0，先把 HIVE_WEBRTC_DOWNLINK_GAIN 降到 1.0-1.5，并确认 4010 已重启使 HIVE_TTS_EDGE_VOLUME 生效；不要同时叠多层增益。
  3. 若 media 仍不干净，走混合下行：WebRTC 只保留上行和连接，AI TTS 下行回到 voice_stream + expo-audio 文件播放。预计这是最接近“对讲同款响亮清晰”的低风险路径。
  4. 不建议短期做 native ADM fork：除非必须维持纯 WebRTC 下行，否则 patch react-native-webrtc/JavaAudioDeviceModule 的 AudioTrack usage/audio attributes 是设备敏感 native 活， 风险高于混合下行。
