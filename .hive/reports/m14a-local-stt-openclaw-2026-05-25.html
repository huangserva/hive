1.1 本地自动检测顺序

src/media-understanding/runner.ts 的 resolveLocalAudioEntry() 是关键：先尝试 sherpa-onnx-offline，再尝试 whisper-cli，最后尝试 Python whisper。只有本地 CLI 都不可用时，才进入 Gemini CLI / provider keys 路径。

resolveLocalAudioEntry()
  ├─ resolveSherpaOnnxEntry()
  ├─ resolveLocalWhisperCppEntry()
  └─ resolveLocalWhisperEntry()

resolveAutoEntries(capability="audio")
  ├─ active configured model
  ├─ local audio CLI
  ├─ gemini CLI
  └─ provider keys: OpenAI → Groq → Deepgram → Google → Mistral

1.2 三个本地 CLI 后端

后端openclaw 检测条件命令形态优点坑
sherpa-onnx-offline PATH 上有二进制；SHERPA_ONNX_MODEL_DIR 下有 tokens.txt、encoder.onnx、decoder.onnx、joiner.onnx sherpa-onnx-offline --tokens=... --encoder=... --decoder=... --joiner=... {{MediaPath}} 真正离线；推理通常比 Python Whisper 更轻；适合长期做本地模型后端。 模型目录结构复杂；中文模型选择和安装 UX 需要单独设计；当前 openclaw skills 里现成的是 sherpa TTS，不是 STT 安装器。
whisper-cli（whisper.cpp） PATH 上有 whisper-cli；优先 WHISPER_CPP_MODEL，否则尝试默认 /opt/homebrew/share/whisper-cpp/for-tests-ggml-tiny.bin whisper-cli -m MODEL -otxt -of {{OutputBase}} -np -nt {{MediaPath}} 无 key、跨平台、模型可控；Node 侧只需子进程；适合 HippoTeam MVP。 需要用户安装 whisper.cpp 和模型；tiny 默认质量不适合中文正式使用；需要转码到 CLI 可接受格式。
Python whisper PATH 上有 whisper whisper --model turbo --output_format txt --output_dir {{OutputDir}} --verbose False {{MediaPath}} 安装路径最容易理解；模型自动下载到 ~/.cache/whisper；多语言/中文能力成熟。 Python/torch 依赖重；首次下载慢；CPU 上 medium/large 延迟明显；要约束 timeout。

1.3 CLI adapter 怎么取输出

runner.entries.ts 对 CLI 做了两层兼容：先按命令识别输出文件，再 fallback 到 stdout。

  • whisper：根据 --output_dir + --output_format txt 找 <basename>.txt。
  • whisper-cli：根据 -otxt + -of {{OutputBase}} 找 {{OutputBase}}.txt。
  • sherpa-onnx-offline：优先从 stdout 的 JSON / quoted JSON 中提取 text，否则用 stdout。
  • 所有 CLI 运行在临时目录 openclaw-media-cli-* 下，结束后清理。

2. 依赖、模型、离线性、中文和性能

问题结论对 HippoTeam 的含义
是否免费 / 无 key 本地 CLI 路线是无 key；模型在本机推理。provider 路线 OpenAI/Groq/Deepgram/Mistral 仍需要 key。 Q10 选项 D 可以绕开飞书免费版限制和外部 ASR 数据出境。
是否真离线 whisper-cli 和 sherpa-onnx-offline 模型就位后可离线；Python whisper 首次可能下载模型，之后离线。 需要做“模型未安装”的明确错误提示，而不是静默失败。
模型体积 Whisper 系列从 tiny/base 到 medium/large 体积差距很大；tiny/base 轻但中文质量有限，small/medium 更稳但慢。sherpa 取决于选择的 ONNX 模型包。 MVP 不应强行下载大模型；应让 user 配置 model path 或使用已装 CLI。
中文支持 Whisper 多语言含中文，实践上 small/medium 比 tiny 更可靠；sherpa 中文取决于具体 ASR 模型。 飞书短语音命令场景可以先用 Whisper small/base 试；重要的是 E2E 量真实口音/噪音。
延迟 openclaw 的本地路线是 batch transcription，不是 realtime streaming。短语音可接受；长音频 CPU 推理会慢。 符合飞书 voice command MVP：用户发语音后等几秒转写，再注入 orch；不适合实时通话。
跨平台 openclaw 用 PATH + env 检测，避免 Node native binding。跨平台负担转移到用户安装 CLI。 HippoTeam 也应先用子进程 CLI，暂不引 native npm binding。
License openclaw 没把本地 STT 模型 vendored 到 repo；它只调用用户安装的 CLI / 模型。具体二进制和模型 license 取决于用户选择。 如果未来做自动下载，需要逐个模型确认 license；MVP 手动配置可降低发布责任。

4. 对 M14a 的具体落地方案

推荐路线：把 Q10 选项 D 作为下一刀：保留当前飞书内置 ASR 实现，但加一个可配置本地 STT provider。默认不自动下载模型；如果检测到本机已有 whisper-cli / whisper，就优先走本地；否则继续走飞书内置或提示 Q10 未决。

4.1 最小接口

interface LocalSttProvider {
  transcribe(input: {
    audioPath: string
    mime?: string
    timeoutMs: number
  }): Promise<{ text: string; provider: string; model?: string }>
}

Feishu transport 已经能下载 audio resource；本地 STT 只需要把 Buffer 落到临时文件，然后调用 CLI provider，成功后复用现有 handleFeishuInbound。

4.2 建议 fallback 顺序

  1. 显式配置的本地 STT 命令：用户完全掌控模型。
  2. 自动检测 whisper-cli + WHISPER_CPP_MODEL。
  3. 自动检测 Python whisper（默认 base 或 small，不要默默用 huge model）。
  4. 飞书内置 speech_to_text.fileRecognize（如果权限可用）。
  5. 外部 ASR provider（只有 user 明确拍板才接）。

4.3 不建议第一刀直接做的事

  • 不建议引入 native npm STT binding：跨平台安装和测试会变复杂。
  • 不建议自动下载大模型：会让用户不清楚发生了什么，也会拖慢首次运行。
  • 不建议把 openclaw 整个 media understanding 系统搬进来：HippoTeam 当前只需要 Feishu voice batch STT。
  • 不建议用 Realtime API 替代：那是另一个产品形态，适合 M14 后续实时语音，不是飞书语音消息 MVP。

5. 风险清单

风险触发条件缓解
模型不存在 / 首次下载 用户只有 whisper 命令但无模型缓存。 日志明确提示；文档给安装命令；不在 runtime 里偷偷下载。
中文识别差 tiny/base 模型、噪音、口音、飞书压缩音频。 E2E 采样；允许配置 model；对中文建议 small/medium。
CPU 占用 / 超时 长语音或大模型在 CPU 上推理。 限制飞书语音时长/文件大小；默认 timeout；失败只 drop 本条。
音频格式不兼容 飞书下载得到 opus/amr，CLI 不能直接读。 必要时在本地 STT 前用 ffmpeg 转 wav；openclaw 已有 ffmpeg helper 思路可借，但 HippoTeam 要另行实现。
安全边界 CLI command 配置可执行任意程序。 MVP 只允许内置白名单命令 whisper-cli / whisper；高级自定义再加确认。
